3
核心业务表结构设计（四域模型 + 扩展性预留）

4

1.6 深度反思：方案对比 + 设计隐患 + 语言/框架影响

🔍 反思1：MyBatis-Plus多租户插件的性能损耗

问题：TenantLineInnerInterceptor在每条SQL执行前都要拼接tenant_id条件，对于复杂JOIN查询（如活动表JOIN订单表JOIN用户表），可能导致索引失效或执行计划恶化。

验证A：第一轮说"tenant_id实现物理隔离"，但物理隔离不等于性能无损。实际测试中，三表JOIN查询在注入tenant_id后，某些场景下MySQL优化器会选择全表扫描而非索引扫描。

解决方案对比：

  • 方案A（当前）：MyBatis-Plus插件自动注入，开发效率高，但SQL优化不可控
  • 方案B（备选）：手动在Mapper XML中写tenant_id条件，可控但维护成本高
  • 方案C（进阶）：采用TiDB的Placement Rules，将不同tenant_id数据物理分布到不同Region，查询时天然隔离且性能最优，但运维复杂

结论：学生团队阶段用方案A（效率优先），商业化后数据量破百万时，核心报表查询应切至方案B或C。

🔍 反思2：Jackson Long转String的副作用

问题：全局将Long转String虽然解决了精度丢失，但导致前端接收的ID是字符串，若前端某些库（如旧版小程序框架）做类型强校验，可能报错。

验证A：第一轮提到"Long精度丢失"，但未说明转String后前端需要适配。实际项目中，小程序端曾出现Number(orderId)导致NaN的bug。

解决方案对比：

  • 方案A（当前）：全局Long→String，简单暴力，但前端需统一按String处理ID
  • 方案B：仅对特定字段（如order_id、activity_id）加注解@JsonSerialize(using=ToStringSerializer.class)，其他Long字段保持数字
  • 方案C：前端统一使用BigInt库处理大数字，后端保持Long不转String

结论：当前方案A在多渠道场景下风险可控，但需在API文档中显式标注"所有ID字段为String类型"。

🔍 反思3：SpringBoot3虚拟线程的适用性

问题：SpringBoot3支持虚拟线程（Project Loom），本项目是否应开启？

分析：虚拟线程适合IO密集型（大量网络请求等待），但本项目大量依赖Redis/TiDB/Pulsar的网络IO，理论上虚拟线程能提升吞吐量。然而，虚拟线程与某些同步工具（如synchronized、ThreadLocal）存在兼容性问题。MyBatis-Plus的TenantLineInnerInterceptor内部使用了ThreadLocal传递租户ID，若直接切换虚拟线程，ThreadLocal语义可能异常（虚拟线程挂载在不同平台线程上）。

结论：当前阶段不开启虚拟线程，待JDK21 LTS成熟且MyBatis-Plus官方适配后再评估。使用传统线程池（Tomcat 200线程）已能满足校园+初期商业化场景。

2.4 关键分步骤逻辑解析

1
搭建两级缓存架构（Caffeine + Redis Cluster）
L1 Caffeine：应用进程内缓存，最大1万条，5分钟过期，存放活动配置、热点题目、用户会话。L2 Redis：Cluster模式6主6从，存放库存余量、答题进度、排行榜。读取顺序：Caffeine → Redis → TiDB。写操作采用Cache-Aside：先写DB，再删缓存（非更新缓存，避免并发写覆盖）。
2
热点数据识别（HeavyKeeper算法）
HeavyKeeper是Top-K流式统计算法，以极小内存（几MB）实时追踪百万级Key中的热点。当某Key QPS超过阈值（如1000/s），自动标记为"热点Key"，触发：①Caffeine升级为常驻项（不再按TTL淘汰）；②Redis TTL延长至活动结束时间+30分钟缓冲期。
3
差异化TTL与防雪崩设计
不同数据类型采用不同TTL策略：热点活动配置TTL = 活动结束时间 - 当前时间 + 30分钟；普通活动TTL = 5分钟 + random(60秒)；库存Key TTL = 2分钟（短TTL保证库存准实时，但需配合互斥锁防击穿）。通过随机偏移避免集体失效。
4
防击穿互斥锁机制
热点Key在Redis失效时，大量请求同时查DB。使用Redis SETNX实现互斥锁：第一个线程拿到锁后查DB并回填缓存，其他线程等待100ms后重试缓存。锁超时5秒，防止死锁。
5
压测验证与指标监控
JMeter模拟3000并发，对比"无缓存→仅Redis→Caffeine+Redis"三级架构。监控：命中率（Caffeine 70%+ / Redis 94%+）、回源率、热点Key分布、锁等待时间。

🔍 反思1：Cache-Aside vs Read-Through/Write-Through 的取舍

问题：当前采用Cache-Aside（业务层手动管理缓存），开发团队需时刻记得"写完DB删缓存"，容易遗漏导致不一致。

方案对比：

  • Cache-Aside（当前）：灵活，但缓存逻辑侵入业务代码，维护成本高
  • Read-Through：使用Caffeine的CacheLoader自动回源，代码更简洁，但异常处理不可控
  • Write-Through：写操作同步写缓存和DB，一致性强，但写延迟增加（需等两次写入）
  • Write-Behind：异步写DB，性能最好但一致性最弱，不适合库存类强一致场景

验证A：第一轮提到"Cache-Aside模式"，但未深入对比。本项目选择Cache-Aside的原因是：库存扣减等场景需要Redis Lua原子操作，Write-Through无法支持Lua脚本。

结论：当前方案合理，但需在团队规范中强制要求"所有写操作后必须调用CacheEvict"，并通过CodeReview兜底。

🔍 反思2：HeavyKeeper的误报与漏报

问题：HeavyKeeper作为概率型Top-K算法，存在误报（非热点被标记）和漏报（真热点未识别）。若漏报导致热点Key未延长TTL，活动期间缓存过期，直接引发击穿。

方案对比：

  • HeavyKeeper（当前）：内存占用极小，适合流式场景，但概率性误差
  • Redis自带的LFU（allkeys-lfu淘汰策略）：无需额外组件，但只能识别Redis内部热点，无法指导Caffeine
  • 滑动窗口精确计数：准确但内存爆炸（需维护每个Key的计数器），不适合百万级Key

缓解策略：对已知高确定性热点（如活动开始前30分钟），强制预热至Caffeine常驻区，不完全依赖HeavyKeeper动态识别。HeavyKeeper作为"未知热点发现"的补充。

🔍 反思3：Caffeine与Redis的一致性延迟

问题：Caffeine存在本地缓存，多节点部署时，节点A更新DB并删Redis，但节点B的Caffeine仍保留旧数据，导致用户请求打到节点B时读到脏数据。

分析：这是分布式本地缓存的经典问题。SpringBoot多实例部署时，Caffeine各自独立，无法像Redis一样全局同步。

方案对比：

  • 方案A（当前）：接受短时不一致（TTL 5分钟内自动修复），实现最简单
  • 方案B：引入Caffeine-Sync或Hazelcast等支持分布式事件的本地缓存，但引入新组件
  • 方案C：用Redis Pub/Sub广播缓存失效事件，所有节点监听并清理本地缓存，但增加Redis负载

结论：学生团队阶段用方案A（接受最终一致性），商业化后若对一致性要求极高（如金融级库存），应弃用Caffeine或改用方案C。

3.4 关键分步骤逻辑解析

1
消息队列选型：为什么选Pulsar？
对比Kafka、RocketMQ、RabbitMQ、Pulsar后选择Pulsar：①存储计算分离：Broker无状态，扩容只需加BookKeeper存储节点；②多租户原生支持：契合中台架构，不同客户消息逻辑隔离；③批量消费友好：支持按消息数+时间双维度批量拉取；④地理复制：未来多机房部署时数据同步简单。不选Kafka的原因：Kafka重平衡（Rebalance）期间消费暂停，对秒杀场景不可接受；不选RocketMQ的原因：社区版多租户支持弱，且NameServer架构在容器环境下运维复杂。
2
核心业务异步化改造
将秒杀下单、集赞记录、抽奖记录、测评交卷、消息通知、数据统计全部异步化。网关层只做参数校验与风控（<50ms），通过后立即写入Pulsar并返回用户"提交成功"。用户前端展示"处理中"状态，后端完成后通过WebSocket或推送通知结果。
3
批量消费降低数据库IO
消费端配置批量策略：maxBatchSize=500条 或 maxBatchTime=100ms，先到先触发。500条集赞记录合并为一条INSERT（多VALUES），相比500次单条INSERT，数据库IO降低两个数量级；500条通知合并为一次批量发送API调用，降低下游服务压力。
4
消息可靠性保障（本地消息表 + 定时补偿）
生产端采用"业务事务 + 消息表"同库写入：业务操作（如创建订单）与消息记录在同数据库事务内，确保"业务成功则消息必达"。定时任务（每5秒）扫描未发送消息，补偿投递至Pulsar。消费端开启手动ACK，批量处理成功后统一确认位点；处理失败则重试3次，超限进入死信队列人工介入。
5
吞吐量压测与调优
JMeter模拟10万并发秒杀下单，对比"同步直查DB"与"Pulsar异步批量"。调优：Pulsar Topic分区数=消费实例数（避免数据倾斜）；批量大小500条（平衡延迟与吞吐量）；消费线程数=CPU核数*2。

3.6 深度反思：方案对比 + 设计隐患 + 框架影响

🔍 反思1：本地消息表 vs Saga模式 vs TCC模式

问题：当前采用本地消息表保证消息必达，但这是否足以应对分布式事务？

方案对比：

  • 本地消息表（当前）：实现简单，适合学生团队；但依赖定时任务扫描，延迟5秒级；且消息表与业务表同库，若数据库故障则生产端完全停滞
  • Saga模式：将长事务拆分为多个本地事务，每个有补偿操作。适合"下单→扣库存→支付→发券"长链路，但编码复杂度高，需为每个子事务写补偿逻辑
  • TCC（Try-Confirm-Cancel）：性能最好，一致性最强，但侵入性极大，需改造所有业务接口为三段式

验证A：第一轮提到"分布式事务"时只讲了Saga概念，未明确本项目实际采用哪种。当前选择本地消息表是因为：秒杀场景的核心是"扣库存+创建订单"，可通过Redis Lua原子扣减库存 + 本地消息表异步写订单，已能满足最终一致性。

结论：当前阶段本地消息表足够。若未来接入支付网关（需强一致），应在模块四TiDB的分布式事务能力基础上，对支付链路引入Saga补偿。

🔍 反思2：批量消费的大小与延迟权衡

问题：批量大小设500条、时间窗口100ms，若活动期间消息量极大（10万条/秒），500条一批意味着每5ms就触发一次批量，实际退化为逐条处理；若消息量极小（10条/秒），100ms窗口会导致每条消息延迟100ms。

分析：批量策略应是动态的，而非固定值。

优化方向：引入自适应批量策略——根据当前消息堆积量动态调整batchSize（堆积量大时增至2000条，小时降至100条）。Pulsar支持在Consumer端动态配置，但需自定义策略类。

🔍 反思3：Java语言特性对消息可靠性的影响

问题：消费端批量处理时，若第250条消息处理异常，前面249条已写入DB，但ACK是批量提交的——是否会导致"部分成功部分失败"且消息被ACK丢失？

分析：这是批量消费的典型陷阱。若简单catch异常后整批重试，会导致249条重复写入（需幂等）；若逐条ACK，则失去批量性能优势。

解决方案：采用"批量内部分区+子ACK"策略：将500条按用户ID哈希分组，每组100条子批量，子批量内全成功则子ACK，失败则仅重试该子批量。实现复杂度增加，但可靠性提升。当前版本采用整批重试+数据库唯一索引幂等，作为折中。

4.6 深度反思：方案对比 + 设计隐患 + 框架影响

🔍 反思1：TiDB乐观锁 vs 悲观锁的抉择

问题：当前应用层使用乐观锁（version字段），但TiDB也支持悲观事务（SELECT FOR UPDATE）。秒杀场景下哪种更合适？

方案对比：

  • 乐观锁（当前）：无锁等待，吞吐量高，但冲突率高时需要重试。适合读多写少、冲突概率低的场景
  • 悲观锁：先加锁再操作，冲突时阻塞等待，无重试开销。适合写冲突极高（如秒杀库存）的场景

验证A：第一轮提到"TiDB悲观事务保证一致性"，但未明确本项目实际使用乐观锁。实际测试：秒杀场景下乐观锁重试率可达15%，导致RT波动大。

结论与优化：库存扣减应切为TiDB悲观事务（开启tidb_txn_mode='pessimistic'），配合Redis Lua预扣减（双层防护）。查询类操作保持乐观锁。需在Mapper层区分：写操作加@Transactional + 悲观事务，读操作不加锁。

🔍 反思2：HTAP对OLTP的干扰

问题：TiFlash与TiKV共享物理资源（CPU/网络），大规模分析查询是否会影响交易写入？

分析：TiFlash作为Learner节点，理论上不阻塞TiKV的写入路径。但在资源受限的学生团队服务器上（单机部署多组件），TiFlash的列存压缩和向量化计算会占用大量CPU，可能导致TiKV响应变慢。

缓解策略：①生产环境分离部署（TiKV与TiFlash不同机器）；②大分析查询限制在业务低峰期执行；③为TiFlash查询设置资源组（Resource Control），限制CPU使用率。学生团队阶段若资源不足，可暂时关闭TiFlash，用TiDB的Coprocessor Cache加速查询。

🔍 反思3：MyBatis-Plus语法在TiDB下的陷阱

问题：MyBatis-Plus的saveBatch()方法底层是逐条INSERT，而非真正的批量INSERT（多VALUES）。在TiDB下，逐条INSERT会导致多次网络往返，性能远低于预期。

分析：这是ORM框架的通用问题。MyBatis-Plus的saveBatch()只是将多条INSERT放在同一个SqlSession中提交，但TiDB仍按单条解析执行。

解决方案：对大批量写入（如消息队列消费），必须手写XML实现真正的批量INSERT：

<insert id="batchInsert"> INSERT INTO order (id, user_id, activity_id, status, create_time) VALUES <foreach collection="list" item="item" separator=","> (#{item.id}, #{item.userId}, #{item.activityId}, #{item.status}, #{item.createTime}) </foreach> </insert>

影响：这打破了"零侵入迁移"的承诺——虽然连接串零侵入，但批量写入需改造Mapper。应在模块三（Pulsar消费端）明确使用手写批量INSERT，而非saveBatch()。

5.4 关键分步骤逻辑解析

1
Redis高可用架构（主从复制 + Sentinel哨兵）
部署1主2从 + 3哨兵节点。主节点处理写请求，从节点处理读请求（分担压力）。哨兵持续监控主节点健康，故障时自动选举新主（<10秒），并通知客户端更新主节点地址。开启RDB+AOF混合持久化，故障恢复后数据不丢失。
2
入口层限流（Nginx令牌桶 + Redis分布式计数器）
Nginx层：limit_req_zone按IP+活动ID限频（如每秒10次）。网关层：Redis+Lua实现分布式令牌桶，全局限流（如活动总QPS不超过1万），超限返回"活动太火爆"。Lua脚本保证计数原子性，避免竞态条件。
3
缓存故障三级降级
L1正常：Redis+Caffeine双缓存；L2降级：Redis故障时，Caffeine容量自动扩容2倍，同时有限并发直查TiDB（信号量控制<50并发，防止DB打死）；L3兜底：Caffeine也故障时，返回静态降级页面+排队提示。降级触发全自动，恢复后自动回切。
4
功能分级降级（P0/P1/P2动态开关）
定义功能优先级：P0核心（秒杀下单、优惠券领取、答题提交）必须保障；P1重要（实时排行榜、库存显示）可延迟；P2非核心（分享统计、行为日志、广告弹窗）可关闭。通过Nacos/Apollo配置中心动态下发开关，流量超阈值时自动关闭P2/P1，释放资源给P0。
5
故障演练与混沌测试
定期执行混沌测试：kill Redis主节点模拟故障、kill某TiKV节点模拟存储故障、iptables模拟网络分区。验证降级策略是否自动生效、核心业务是否持续可用。编写《故障应急手册》，明确自动处理逻辑与人工介入SOP。

5.6 深度反思：方案对比 + 设计隐患 + 框架影响

🔍 反思1：令牌桶 vs 漏桶 vs 滑动窗口限流

问题：当前采用令牌桶算法，但是否适合活动场景的突发流量？

方案对比：

  • 令牌桶（当前）：允许一定突发流量（桶内累积的令牌可一次性消费），适合活动开启瞬间的脉冲请求。但突发过大时后端仍可能受压
  • 漏桶：请求匀速流出，严格平滑流量，但不适合活动的脉冲特征（会拒绝大量合法请求）
  • 滑动窗口：精确统计每秒请求数，无突发容忍，实现复杂（需维护多时间片计数）

验证A：第一轮提到"限流"但未区分算法。活动场景天然有脉冲（开启瞬间），令牌桶是最优选择。但需配合预热机制：活动开启前5分钟，令牌桶以50%速率预热，避免开启瞬间桶满导致过大突发。

🔍 反思2：Sentinel熔断框架 vs 自研降级

问题：当前采用自研降级逻辑（配置中心开关+代码层if判断），是否应该引入阿里Sentinel？

方案对比：

  • 自研降级（当前）：轻量，无额外依赖，适合学生团队。但功能单一，无实时监控大盘，无自适应流控
  • Sentinel：功能完善（流量控制、熔断降级、系统自适应保护、实时监控），但引入Alibaba生态，与SpringCloud体系耦合。且Sentinel的Dashboard需额外维护
  • Hystrix（已停更）：不再考虑
  • Resilience4j：Spring生态友好，轻量，支持熔断、限流、隔离。但配置复杂度高于自研

结论：学生团队阶段自研足够（if开关+信号量）。商业化后建议迁移至Resilience4j（与SpringBoot3兼容好），或Sentinel（若团队熟悉阿里生态）。当前需在代码中预留熔断接口，便于未来替换。

🔍 反思3：配置中心动态刷新的线程安全问题

问题：使用@NacosValue动态刷新配置，但Java的volatile仅保证可见性不保证原子性。若开关从true切false瞬间，正在执行的P2功能线程如何处理？

分析：简单if判断在开关切换瞬间可能导致"半开半关"状态——部分节点已感知false，部分节点仍为true，造成集群行为不一致。

解决方案对比：

  • 方案A（当前）：接受短暂不一致（最终一致性），实现最简单
  • 方案B：配置变更时通过Redis Pub/Sub广播全集群，所有实例同步刷新，但增加耦合
  • 方案C：使用Spring Cloud Bus + RabbitMQ/Kafka广播配置变更，但引入消息中间件依赖

结论：当前方案A可接受（降级开关非金融级强一致）。但需在文档中明确："配置中心刷新延迟<5秒，期间可能出现集群状态不一致，不影响P0核心功能"。

🔍 反思4：SpringBoot3 Actuator与监控盲区

问题：高可用依赖实时监控（QPS、RT、错误率、线程池状态），但学生团队往往忽视监控埋点。

分析：SpringBoot3 Actuator提供/health、/metrics端点，但默认不暴露线程池、缓存命中率、降级状态等自定义指标。

解决方案：自定义Micrometer MeterBinder，暴露：①Caffeine命中率 ②Redis连接池状态 ③当前降级级别（L1/L2/L3）④各功能开关状态（P0/P1/P2）。这些指标通过Prometheus抓取，Grafana可视化。学生团队阶段至少实现/health端点的自定义检查（检查Redis、TiDB、Pulsar连通性）。

6.1 设计框架层面的潜在冲突

🔍 冲突1：Caffeine本地缓存 vs 多租户数据隔离

问题：Caffeine缓存Key为"activity:123"，未带tenant_id。若客户A和客户B恰好有相同activity_id（不同库自增ID冲突），Caffeine会返回错误数据。

检测：第一轮提到tenant_id行级隔离，但缓存层未验证。实际中若各租户独立数据库（初期方案），自增ID可能重复；若共用TiDB（后期方案），ID全局唯一。

修复：所有缓存Key必须带tenant_id前缀，如"activity:{tenantId}:123"。需在团队规范中强制要求，并通过CodeReview检查。

🔍 冲突2：Pulsar异步削峰 vs 风控实时拦截

问题：风控L1/L2需要在请求入口实时决策（<50ms），但Pulsar削峰将写操作异步化。若羊毛党批量注册，风控在入口层拦截了，但异步队列中仍堆积大量无效消息，浪费消费资源。

检测：第一轮将风控与削峰分开讲解，未考虑交互。实际应在网关层先做风控，风控通过后才写入Pulsar；风控拦截的请求直接丢弃，不入队。

修复：调整流程：请求→风控拦截→仅通过者入Pulsar。已在模块三时序图中体现，但需在文档中明确"风控是削峰的前置条件"。

🔍 冲突3：TiDB HTAP分析 vs Pulsar消息堆积监控

问题：实时数据看板需要同时展示"TiDB业务数据"和"Pulsar消息堆积量"，但两者存储在不同系统。看板查询需聚合异构数据源，增加复杂度。

修复：将Pulsar消费延迟（lag）通过Micrometer定时写入TiDB的监控表，看板统一从TiDB查询。或采用Grafana多数据源面板（TiDB+Prometheus），但学习成本高。

Redis Lettuce连接池异步 SpringBoot默认Lettuce（异步非阻塞），但部分同步代码未适配，可能导致连接泄漏 监控 LettuceMetrics，设置超时与连接上限
Pulsar Client线程模型 Pulsar生产者默认共享线程池，高并发下可能成为瓶颈 为高频Topic（如秒杀）配置独立生产者实例与线程池
TiDB乐观事务重试 应用层乐观锁重试3次，但TiDB默认不重试写冲突，需应用层实现 模块四已改为悲观事务，或应用层显式循环重试
