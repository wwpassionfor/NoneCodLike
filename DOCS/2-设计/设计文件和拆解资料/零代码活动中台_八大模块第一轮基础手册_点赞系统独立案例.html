🎯 决策4：AI风控深度 → 选B（Flink CEP + 特征工程）

以下为三个选项的完整技术解析与综合对比，供评审/面试时引用：

维度方案A：规则引擎（Drools/Aviator）方案B：Flink CEP + 特征工程（本项目采用）方案C：深度学习（LSTM/神经网络）
技术原理 基于预定义规则（阈值、模式匹配）进行硬编码判断。如"IP 1分钟内请求>100次→拦截"。 基于Flink流处理引擎，用CEP（Complex Event Processing）定义事件模式序列（如"登录→领券→下单"在5秒内完成→可疑），结合简单特征工程（答题时长分布、IP聚集度、设备指纹相似度）打分。 基于神经网络（如LSTM行为序列分类、GNN团伙检测）自动学习作弊模式，无需人工定义规则。
实现复杂度 ⭐ 低。引入Drools/Aviator依赖，写规则文件（drl/脚本），Java直接调用。 ⭐⭐⭐ 中。需部署Flink集群，编写CEP Pattern，设计特征工程Pipeline，维护模型版本。 ⭐⭐⭐⭐⭐ 高。需数据标注、模型训练（GPU）、调参、部署推理服务（TensorFlow Serving/TorchServe），持续迭代。
响应延迟 <1ms（内存中规则匹配） 百毫秒级（Flink窗口聚合+CEP匹配），可满足实时拦截需求 十毫秒~秒级（取决于模型复杂度），需独立推理服务
可解释性 极高。规则透明，可直接向用户解释"因IP超限被拦截"。 中等。CEP模式可解释（"因5秒内完成3个操作"），但特征工程权重需可视化。 低。黑盒模型，难以解释"为什么判定作弊"，合规场景受限。
对抗新型攻击 弱。规则需人工更新，新型攻击手段（如慢速刷票、分布式代理）难以覆盖。 较强。通过行为序列模式（如"固定间隔点击"）和统计特征（如"答题时长标准差异常"）可捕获未知模式。 强。模型自动学习行为分布，理论上可识别任意模式，但依赖训练数据覆盖度。
资源消耗 CPU/内存消耗极低，单节点可支撑万级QPS。 中等。Flink集群需2-4节点（JobManager+TaskManager），内存占用取决于窗口大小。 高。训练需GPU，推理需独立服务集群，维护成本指数级增长。
学生团队可驾驭性 极易。1-2天可上手，适合MVP验证。 可驾驭。Flink有完善文档和社区，CEP API成熟；特征工程无需高数基础，统计指标即可。 困难。需深度学习背景、数据标注团队、模型运维经验，学生团队难以闭环。
与本项目契合度 可作为L1规则层（已融入模块五），但单独作为AI风控主力太单薄。 最佳平衡点：技术深度足够体现架构能力（流计算+模式识别），工程复杂度可控，与Pulsar/TiDB技术栈天然集成（Flink消费Pulsar，结果写入TiDB）。 过度设计。校园活动/中小企业营销场景的作弊模式相对简单，深度学习ROI极低。
典型应用场景 固定阈值场景：IP限频、账号限次、黑名单拦截。 行为序列异常：秒杀机器人（固定间隔下单）、测评代考（答题时长分布异常）、团伙刷票（IP/答案聚集）。 金融反欺诈（交易行为建模）、大型平台级风控（抖音/淘宝级体量）。
风险与局限 规则僵化，误杀率高（如校园WiFi下多人共用IP被误拦），维护成本高（每新增攻击手段需改代码发版）。 CEP模式设计需业务理解（如"切屏3次"模式定义）；窗口设置不当导致延迟过高；特征工程需持续调优。 冷启动问题（新活动无历史数据）；模型漂移（作弊手段进化后准确率下降）；解释性不足导致客诉。

模块二多级缓存 + 热点数据治理

解决：峰值崩溃 解决：接口响应慢 解决：缓存击穿/雪崩

2.1 问题背景

📍 校园场景

校庆秒杀3000人同时查库存，MySQL连接池打满，RT从200ms飙至15秒；所有缓存TTL相同导致集体过期，数据库CPU 100%假死。

2.3 分步骤逻辑

1
搭建两级缓存架构
L1 Caffeine：应用进程内，最大1万条，5分钟过期，读多写少数据。L2 Redis：Cluster模式，库存/进度/排行榜。读取顺序：Caffeine→Redis→TiDB。
2
热点数据识别（HeavyKeeper）
Top-K流式统计算法，极小内存实时追踪热点Key。QPS超阈值（1000/s）自动标记，TTL延长至活动结束，升级为Caffeine常驻项。
3
差异化TTL与防雪崩
热点Key TTL=活动结束+30分钟缓冲；普通Key TTL=5分钟+random(60秒)；库存Key TTL=2分钟（短TTL保证准实时）。
4
防击穿互斥锁
Redis SETNX实现互斥锁：第一个线程查DB并回填，其他线程等待100ms后重试。锁超时5秒防死锁。
5
压测验证
JMeter模拟3000并发，对比"无缓存→仅Redis→Caffeine+Redis"三级架构RT与吞吐量。

3.3 分步骤逻辑

1
消息队列选型：Pulsar
存储计算分离、多租户原生支持、批量消费友好、地理复制。不选Kafka（重平衡暂停消费）、不选RocketMQ（社区版多租户弱）。
2
核心业务异步化改造
秒杀下单、集赞记录、抽奖记录、测评交卷、消息通知、数据统计全部异步化。网关只做参数校验与风控（<50ms），立即返回"提交成功"。
3
批量消费降低数据库IO
maxBatchSize=500条 或 maxBatchTime=100ms。500条集赞记录合并为一条INSERT（多VALUES），IO降低两个数量级。
4
消息可靠性保障
本地消息表+定时补偿：业务操作与消息写入同库事务，确保"业务成功则消息必达"。消费端手动ACK，失败重试3次，超限入死信队列。
5
吞吐量压测与调优
JMeter模拟10万并发秒杀下单。Topic分区数=消费实例数；批量大小500条；消费线程数=CPU核数*2。

4.3 分步骤逻辑

1
技术选型与迁移评估
TiDB兼容MySQL协议，MyBatis-Plus零改代码；自动分片无需指定分片键；HTAP同时服务OLTP+OLAP；存储计算分离扩容加TiKV节点即可。
2
无缝迁移与代码零侵入
仅修改JDBC连接串，所有Mapper/XML/Lambda保留。TiDB Lightning工具全量导入历史数据（50万条耗时2小时）。迁移期间双写验证。
3
自动分片与分片键设计
activity表按tenant_id哈希（同一客户数据聚集）；order表按order_id哈希（避免自增ID热点）；user_behavior按create_time范围分片（利于归档）。
4
分布式事务一致性保障
TiDB默认Percolator模型，Snapshot Isolation隔离级别。秒杀"扣库存→创建订单"跨表操作自动保证ACID。极度敏感操作加乐观锁version字段。
5
HTAP实时分析支撑看板
TiFlash列存扩展实时同步TiKV数据（Raft Learner，延迟<1秒）。聚合查询（院系参与率、7天趋势）路由至TiFlash，性能提升10倍+。

6.1 问题背景

📍 秒杀超卖场景

库存只剩1件，用户A和B同时点击"立即抢购"。分布式锁方案下，SETNX获取锁→查库存→扣减→释放锁，整个流程4步，锁超时风险高（网络抖动导致锁未释放，其他用户永远等待）。

📍 优惠券重复领取

同一用户快速双击"领取"按钮，两次请求同时到达，分布式锁未生效前两次都查到"未领取"，结果用户领到2张限领1张的券。

6.2 核心原理：为什么Lua能替代分布式锁？

🔬 Redis Lua原子性保证

Redis是单线程执行模型，Lua脚本在Redis服务端一次性执行完毕，中间不会被其他命令插入。这意味着"查库存→判断→扣减"三步在Redis内部原子完成，无需分布式锁，无锁超时风险，无死锁问题。

6.4 分步骤逻辑

1
Lua脚本设计（库存扣减）
脚本逻辑：GET库存→IF大于0 THEN DECR并返回1 ELSE返回0。整个逻辑在Redis单线程原子执行，天然避免竞态条件。
2
资格校验Lua脚本
抽奖/秒杀前校验用户资格：SISMEMBER检查用户是否在已参与集合中→若否，SADD加入集合并返回成功→若是，返回已参与。原子保证"查+写"不被打断。
3
优惠券核销Lua脚本
原子执行：检查用户领取记录（Hash）→若未超limit则HINCRBY增加计数并返回成功→若超限返回失败。避免"查时未超限，写时超限"的竞态窗口。
4
热Key预处理
活动发布时将库存Key、资格Key预热至Redis，避免首次请求冷启动。对超高并发Key（如爆款商品）采用Redis Cluster分片或本地缓存+Caffeine双重防护。
5
失败重试与异步对账
Lua返回失败后，前端提示"库存不足"或"已参与"。成功请求异步写入Pulsar，最终落库TiDB。每日对账：比对Redis库存余量与TiDB订单记录，差异自动补偿。

6.5 技术对比：Lua原子 vs 其他方案

维度Redis Lua（本项目采用）SETNX分布式锁数据库悲观锁数据库乐观锁
实现复杂度中（需学Lua语法）低（SETNX+过期时间）低（SELECT FOR UPDATE）低（version字段）
性能极高（Redis内存操作+单线程）高（但加锁/释放锁有网络开销）低（数据库行锁阻塞）中（冲突时需重试）

死锁风险无有（未正确释放锁）有（事务未提交）无
适用场景库存扣减、资格校验、计数器复杂业务逻辑（多步操作）强一致金融交易读多写少、冲突率低

关键代码：点赞原子操作

⚙️ Redis Lua脚本（防重复+计数原子化）

// 点赞Lua脚本：检查是否已点赞→未点赞则记录+计数+返回成功 String LIKE_LUA = "local userKey = KEYS[1]; local contentKey = KEYS[2]; local userId = ARGV[1]; " + "if redis.call('SISMEMBER', userKey, userId) == 1 then return 0; end " + // 已点赞 "redis.call('SADD', userKey, userId); " + "redis.call('INCR', contentKey); " + "return 1;"; // 取消赞Lua脚本 String UNLIKE_LUA = "local userKey = KEYS[1]; local contentKey = KEYS[2]; local userId = ARGV[1]; " + "if redis.call('SISMEMBER', userKey, userId) == 0 then return 0; end " + // 未点赞 "redis.call('SREM', userKey, userId); " + "redis.call('DECR', contentKey); " + "return 1;"; // Java调用层 public boolean like(Long userId, Long contentId) { String userKey = "like:user:" + contentId; String contentKey = "like:count:" + contentId; Long result = redisTemplate.execute( new DefaultRedisScript<>(LIKE_LUA, Long.class), Arrays.asList(userKey, contentKey), userId.toString() ); if (result == 1) { // 异步写入Pulsar，最终落库TiDB pulsarProducer.send("like-record", new LikeEvent(userId, contentId, 1)); return true; } return false; // 已点赞或失败 }
