2.3 数据流图

用户发起活动请求（如：参与秒杀） │ ▼ Nginx 反向代理 → 限流检查 → 路由到应用服务 │ ▼ Spring Boot Controller 接收请求 │ ├── Step 1: 查询缓存 │ ├── Caffeine L1 缓存查询（命中 → 直接返回） │ └── Redis L2 缓存查询（命中 → 返回并回填L1） │ ├── Step 2: 热点检测 │ └── HeavyKeeper 算法识别热点Key → 动态调整TTL │ ├── Step 3: 业务逻辑处理 │ ├── 风控校验（防刷、防羊毛党） │ ├── 库存检查（Redis原子操作） │ └── 活动规则验证 │ ├── Step 4: 异步写入 │ ├── Lua脚本原子更新Redis库存 │ └── 发送消息到Pulsar（订单、行为记录） │ └── Step 5: 返回响应 └── 快速返回给用户（<50ms） 异步消费流程： Pulsar Consumer 批量拉取消息 │ ├── 消息去重（按userId+activityId） ├── 分类处理（订单、行为、通知） ├── 批量写入TiDB（500条/批） └── 确认消费（ACK） 定时对账流程： 定时任务（每天凌晨2点） │ ├── 扫描Redis活动数据 ├── 比对TiDB数据 ├── 计算差异 └── 发送补偿事件到Pulsar

3.2 开发规范

📝 命名规范

  • 类名：大驼峰（PascalCase）
  • 方法名：小驼峰（camelCase）
  • 常量：全大写+下划线（UPPER_SNAKE）
  • 包名：全小写+点分隔
  • 数据库表：小写+下划线（snake_case）

🔒 安全规范

  • 所有外部输入必须校验
  • SQL使用预编译语句
  • 敏感信息加密存储
  • API接口鉴权（JWT）
  • 日志脱敏处理

⚡ 性能规范

  • 避免N+1查询
  • 批量操作替代循环
  • 合理使用索引
  • 缓存优先策略
  • 异步处理耗时操作

🧪 测试规范

  • 单元测试覆盖率>80%
  • 集成测试覆盖核心流程
  • 压测验证性能指标
  • 混沌工程验证高可用
  • 自动化CI/CD流水线

⚖️ 四、多方案对比评估

4.1 方案一：传统单体架构（基准方案）

📦 传统单体架构

技术栈：Spring Boot + MySQL + 简单Redis缓存

优势
  • 架构简单，易于理解和维护
  • 开发成本低，适合小型项目
  • 部署简单，单机即可运行
  • 技术栈成熟，社区资源丰富
劣势
  • 峰值QPS仅1000-5000，无法应对大促
  • 单点故障风险高，可用性99.5%
  • 扩展困难，需停机维护
  • 数据库成为瓶颈，无法水平扩展

适用场景：日活<1万的小型活动，预算有限的项目

4.2 方案二：微服务架构（备选方案）

🔧 微服务架构

技术栈：Spring Cloud + Kubernetes + 多数据库

优势
  • 服务独立部署，灵活扩展
  • 技术栈可多样化，按需选择
  • 故障隔离，影响范围小
  • 适合大型团队并行开发
劣势
  • 架构复杂度高，运维成本大
  • 需要K8s集群，基础设施要求高
  • 分布式事务处理复杂
  • 团队需要微服务经验

适用场景：大型企业，多团队协作，日活>100万

4.3 方案三：零代码中台架构（推荐方案）⭐

🚀 零代码中台架构

技术栈：Spring Boot 3 + 两级缓存 + Pulsar + TiDB

优势
  • 峰值QPS 50000+，性能提升10-50倍
  • 零代码配置，活动上线从周缩短到天
  • 三级降级策略，可用性99.99%
  • 自动扩容，无需停机维护
  • Docker一键部署，运维成本低
  • 内置风控，防刷防羊毛党
劣势
  • 技术栈较新，团队需要学习成本
  • 初期搭建复杂度高于单体
  • 需要一定的中间件运维经验

适用场景：中小企业、电商、品牌方，日活1万-100万

📋 五、综合审查报告

5.1 风险评估

🔴 高风险

技术栈学习成本

团队需要掌握Spring Boot 3、Java 21虚拟线程、Pulsar、TiDB等新技术，初期学习曲线较陡。

缓解措施：提供培训文档、代码示例、渐进式迁移

🟡 中风险

中间件运维复杂度

Redis集群、Pulsar、TiDB需要一定的运维经验，故障排查难度高于单体。

缓解措施：Docker一键部署、完善监控告警、自动化运维脚本

🟢 低风险

数据迁移风险

从MySQL迁移到TiDB需要数据同步，可能存在数据不一致。

缓解措施：双写验证、定时对账、灰度切换

🟢 低风险

第三方依赖风险

依赖开源组件，可能存在版本兼容性问题。

缓解措施：锁定版本、定期更新、备份方案

👥 人力资源

  • 后端开发：2-3人（Java/Spring Boot）

  • ROI：活动收入提升3-5倍
