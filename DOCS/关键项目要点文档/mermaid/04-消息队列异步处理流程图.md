# 消息队列异步处理流程图

```mermaid
graph TB
    subgraph 生产者端["Producer - 异步发送"]
        A[用户点赞请求] --> B[执行Lua脚本<br/>原子性操作Redis]
        B --> C{操作成功?}
        C -->|成功| D[构建ThumbEvent事件]
        D --> E[PulsarTemplate.sendAsync]
        E --> F[异步发送到thumb-topic]
        F --> G{发送成功?}
        G -->|成功| H[立即返回给用户 ✅]
        G -->|失败| I[补偿回滚Redis<br/>删除点赞标记]
        I --> J[返回错误 ❌]
        C -->|失败| K[抛出业务异常<br/>"用户已点赞"]
    end
    
    subgraph Broker端["Pulsar Broker - 消息中转"]
        F --> L[thumb-topic 分区存储]
        L --> M[批量积攒消息<br/>maxNumMessages=1000<br/>timeout=10s]
    end
    
    subgraph 消费者端["Consumer - 批量消费"]
        M --> N[ThumbConsumer接收]
        N --> O[按userId+blogId分组]
        O --> P[取每个分组最新事件<br/>奇数次=点赞,偶数次=取消]
        P --> Q{事件类型?}
        Q -->|INCR 点赞| R[batchInsertThumbs<br/>批量插入点赞记录]
        Q -->|DECR 取消| S[批量删除点赞记录]
        R --> T[batchUpdateBlogs<br/>批量更新博客点赞数]
        S --> T
        T --> U{处理成功?}
        U -->|成功| V[ACK确认 ✅]
        U -->|失败| W[NACK重试<br/>延迟1s→2s→4s...最大60s]
        W --> X{重试次数<3?}
        X -->|是| W
        X -->|否| Y[转入死信队列DLQ<br/>人工介入处理]
    end
    
    subgraph 补偿机制["定时对账补偿"]
        Z[每天凌晨2点执行] --> AA[扫描所有用户Redis点赞数据]
        AA --> AB[对比MySQL实际记录]
        AB --> AC{发现差异?}
        AC -->|有差异| AD[发送补偿事件到Pulsar]
        AC -->|一致| AE[对账完成 ✅]
    end
```

## 核心配置参数

### 批量消费配置
- **单批最大消息数**：1000条
- **超时时间**：10秒（凑够1000条或超时都会触发）
- **订阅类型**：Shared（共享订阅，支持并行消费）

### 重试策略配置
**NACK重试（消费者主动拒绝）**：
- 初始延迟：1秒
- 最大延迟：60秒
- 延迟倍数：2（指数退避）

**ACK超时重试（消费者未及时确认）**：
- 初始延迟：5秒
- 最大延迟：300秒（5分钟）
- 延迟倍数：3

**死信队列策略**：
- 最大重试次数：3次
- 死信主题：`thumb-dlq-topic`
