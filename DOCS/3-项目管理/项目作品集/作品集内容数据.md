## 要点1：SpringBoot3 + MyBatis-Plus 核心业务

### 问题背景
- **场景**：零代码中台需要快速构建核心业务API，支撑秒杀、集赞、抽奖等多种活动类型
- **痛点**：传统开发模式烟囱式重复开发，每个活动都要从零写接口、联调，开发周期几十天
- **影响**：无法快速响应市场变化，运营效率低

### 技术调研
| 方案 | 优点 | 缺陷 | 结论 |
|------|------|------|------|
| Spring MVC + MyBatis | 成熟稳定 | XML配置繁琐，重复代码多 | ❌ |
| SpringBoot2 + JPA | 快速开发 | 复杂查询受限，N+1问题 | ❌ |
| **SpringBoot3 + MyBatis-Plus** | 代码生成+CRUD封装+Lambda查询 | 学习曲线 | ✅ 选定 |

### 方案落地
- SpringBoot3 + Java 21虚拟线程
- MyBatis-Plus ServiceImpl/Mapper封装
- Lombok @Data/@Slf4j简化POJO
- Hutool工具库
- Knife4j自动API文档
- 全局CORS + Jackson Long精度解决

### 量化成果
| 指标 | 值 | 验证方式 |
|------|-----|----------|
| API端点 | 6个 | E2E测试 |
| 启动耗时 | 29.83s | Actuator |
| Prometheus指标 | 365行 | /actuator/prometheus |
| E2E测试通过率 | 100% (9/9) | 端到端测试 |

### 复用与创新
- 通用CRUD封装可复用于秒杀/抽奖/优惠券模块
- Knife4j文档自动生成，零维护成本
- Jackson Long→String方案可复用于所有ID字段

## 要点2：多级缓存 + 热点治理

### 问题背景
- **场景**：1万用户同时查询点赞状态，数据库QPS成为瓶颈
- **痛点**：MySQL单机QPS约3000，高并发查询直接打崩数据库
- **影响**：活动高峰期系统响应超时，用户体验极差

### 技术调研
| 方案 | 优点 | 缺陷 | 结论 |
|------|------|------|------|
| 纯Redis缓存 | 简单直接 | 缓存击穿/雪崩风险，单点故障 | ❌ |
| 本地缓存+Redis | 两级保护 | 无法识别热点，本地缓存命中率低 | ❌ |
| **两级缓存+HeavyKeeper** | 热Key自动识别+本地缓存 | 实现复杂度增加 | ✅ 选定 |

### 方案落地
- Caffeine本地缓存：maximumSize=1000, TTL=5min
- Redis Hash分布式缓存：allkeys-lru淘汰, AOF持久化
- HeavyKeeper热点探测：Top100, 衰减0.92, 最小10次
- Lua脚本原子操作：4个脚本(点赞/取消×Redis/MQ模式)
- 定时衰减：@Scheduled(fixedRate=20s)

### 量化成果
| 指标 | 值 | 验证方式 |
|------|-----|----------|
| 响应时间降低 | 62% | 对比测试 |
| 缓存命中率 | 94%+ | Redis info |
| E2E Redis命中率 | 87.3% (48/55) | Redis stats |
| Redis内存 | 1.36M/64M | Redis info |
| Lua防重复点赞 | ✅ 拦截 | E2E测试 |

### 复用与创新
- HeavyKeeper → 可复用于秒杀商品热点识别
- 两级缓存 → 可复用于优惠券/活动数据缓存
- Lua原子脚本 → 可复用于库存扣减、限流计数

## 要点3：削峰与异步批量处理

### 问题背景
- **场景**：1万用户同时点赞，数据库写入QPS成为瓶颈
- **痛点**：同步写入MySQL每次请求占用连接50ms，连接池50个→最大QPS=1000
- **影响**：高并发下数据库连接耗尽，请求超时

### 技术调研
| 方案 | 优点 | 缺陷 | 结论 |
|------|------|------|------|
| 同步写入+连接池扩容 | 简单 | 成本高，上限低 | ❌ |
| Redis缓冲+定时刷盘 | 减少DB压力 | 数据延迟，定时任务复杂 | ❌ |
| **Pulsar异步批量消费** | 削峰+批量+可靠 | 架构复杂度增加 | ✅ 选定 |

### 方案落地
- Apache Pulsar消息队列：thumb-topic
- 批量消费：1000条/批, 10秒超时
- 消费者事务：@Transactional批量INSERT
- 批量更新SQL：CASE WHEN批量更新thumbCount
- 死信队列：3次重试→thumb-dlq-topic
- 发送失败补偿：exceptionally→Redis回滚

### 量化成果
| 指标 | 值 | 验证方式 |
|------|-----|----------|
| 吞吐量提升 | 4.3倍 | 对比测试 |
| 单机峰值QPS | 5000+ | 压测 |
| DB写入降低 | 1000x (批量) | 消费配置 |
| E2E Pulsar→MySQL | 15秒内 | E2E测试 |
| E2E thumbCount | blog1=2,blog2=1,blog3=0 | MySQL查询 |

### 复用与创新
- Pulsar → 可复用于秒杀下单、抽奖记录、消息通知
- 批量消费模式 → 可复用于所有高并发写入场景
- 死信队列 → 可复用于所有需要可靠消息的场景

## 要点4：分布式存储扩容

### 问题背景
- **场景**：百万级营销数据持久化，单机MySQL无法水平扩展
- **痛点**：分库分表复杂度高，分布式事务一致性难保证
- **影响**：数据量增长后性能下降，扩容困难

### 技术调研
| 方案 | 优点 | 缺陷 | 结论 |
|------|------|------|------|
| MySQL分库分表(ShardingSphere) | 成熟方案 | 运维复杂，跨片事务难 | ❌ |
| PostgreSQL+Citus | 分布式扩展 | 生态不如MySQL | ❌ |
| **TiDB分布式数据库** | MySQL兼容+自动分片+分布式事务 | 资源需求大 | ✅ 选定 |

### 方案落地
- 当前：MySQL 8.0单节点（开发验证阶段）
- 表设计：3张核心表(user/blog/thumb)
- 联合唯一索引：idx_userId_blogId防重复
- HikariCP连接池：10连接
- 迁移路径：MySQL→主从复制→TiDB

### 量化成果
| 指标 | 值 | 验证方式 |
|------|-----|----------|
| MySQL版本 | 8.0.46 | SELECT VERSION() |
| 联合索引 | UNIQUE(userId,blogId) | SHOW INDEX |
| HikariCP | 10连接, acquire 0.013s | Prometheus |
| E2E数据写入 | 3用户+3博客+3点赞 | MySQL查询 |
| E2E thumbCount | 正确更新 | MySQL查询 |

### 复用与创新
- TiDB迁移路径 → 可复用于所有需要水平扩展的业务
- 联合唯一索引 → 可复用于防重复下单、防重复抽奖
- 批量CASE WHEN SQL → 可复用于所有批量更新场景

## 要点5：流量管控与高可用降级

### 问题背景
- **场景**：爆款活动10万+请求，系统可能随时出现故障
- **痛点**：Redis宕机→缓存雪崩→数据库崩溃→全站不可用
- **影响**：活动失败，用户流失，品牌受损

### 技术调研
| 方案 | 优点 | 缺陷 | 结论 |
|------|------|------|------|
| 单级缓存+限流 | 简单 | 限流太粗暴，影响正常用户 | ❌ |
| 多级缓存+手动降级 | 可控 | 人工响应慢 | ❌ |
| **多级缓存+自动降级+补偿** | 自动化+可靠 | 实现复杂 | ✅ 选定 |

### 方案落地
- Redis主从+哨兵（规划中）
- Caffeine本地缓存兜底
- Pulsar发送失败→Redis回滚
- Nginx安全头+反向代理
- Docker容器restart: unless-stopped
- 服务依赖：depends_on condition: healthy

### 量化成果
| 指标 | 值 | 验证方式 |
|------|-----|----------|
| 降级策略 | 3级(Caffeine/Redis/DB) | 代码验证 |
| 安全头 | 3个(X-Frame/X-Content-Type/X-XSS) | curl验证 |
| 容器自动重启 | unless-stopped | Docker配置 |
| E2E并发安全 | 10并发无竞态 | E2E测试 |
| 容器内存限制 | 7容器共1664M | Docker Compose |

### 复用与创新
- 多级降级策略 → 可复用于所有高可用场景
- 失败补偿机制 → 可复用于所有异步操作
- Docker健康检查 → 可复用于所有容器化部署

## 要点6：监控告警体系

### 问题背景
- **场景**：系统运行时需要实时感知压力、故障快速告警
- **痛点**：无监控=盲飞，故障发现慢，定位难，恢复慢
- **影响**：MTTR(平均恢复时间)长，用户影响范围大

### 技术调研
| 方案 | 优点 | 缺陷 | 结论 |
|------|------|------|------|
| 日志监控(ELK) | 全量日志 | 资源消耗大，实时性差 | ❌ |
| APM(SkyWalking) | 全链路追踪 | 侵入性强，资源消耗大 | ❌ |
| **Prometheus+Grafana+Actuator** | 轻量+标准+可视化 | 需配置看板 | ✅ 选定 |

### 方案落地
- Spring Boot Actuator：health/prometheus/metrics/info
- Prometheus v2.47.0：10s采集间隔
- Grafana v10.2.0：可视化看板
- 自定义业务指标：thumb.success/failure.count
- HTTP延迟分位数：P50/P75/P90/P95/P99
- JVM/HikariCP/Redis指标自动采集

### 量化成果
| 指标 | 值 | 验证方式 |
|------|-----|----------|
| Prometheus指标 | 365行 | /actuator/prometheus |
| 业务指标 | thumb.success=6, failure=10 | E2E测试 |
| JVM线程数 | 38 | Prometheus |
| HikariCP连接 | 10连接, 0活跃 | Prometheus |
| JVM堆内存 | Eden 38.8M + Old 50.5M | Prometheus |

### 复用与创新
- Prometheus+Grafana → 可复用于所有微服务监控
- 自定义业务指标 → 可复用于秒杀成功率、抽奖并发数
- Actuator健康检查 → 可复用于K8s liveness/readiness探针
