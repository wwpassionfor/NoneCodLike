## 7. 热点缓存优化（HeavyKeeper 算法）

### 问题背景

在缓存系统中，存在"二八定律"：20% 的热点数据承载了 80% 的访问流量。如果不加识别，这些热 Key 会反复穿透 L1 缓存访问 Redis，造成：

**具体问题**：
1. 某篇爆款博客被 10 万用户同时点赞查询
2. 每次查询都访问 Redis（L1 未缓存）
3. Redis 该 Key 的 QPS 飙升到数万
4. 导致 Redis 单线程阻塞，影响其他请求
5. 甚至触发 Redis 雪崩

**业务场景举例**：
- 明星发布动态，粉丝蜂拥点赞
- 热门活动上线，用户疯狂参与
- 营销推广爆款，短时间内爆发流量

### 技术调研

**热点识别算法对比**：

| 算法 | 空间复杂度 | 时间复杂度 | 准确性 | 适用场景 |
|------|------------|------------|--------|----------|
| **HeavyKeeper** ⭐ | O(k×d×w) | O(1) | 高（支持衰减） | **生产环境推荐 ✅** |
| Count-Min Sketch | O(d×w) | O(1) | 中（无衰减） | 统计场景 |
| LFU | O(n) | O(log n) | 准确但内存大 | 小规模数据 |
| 滑动窗口计数器 | O(n) | O(1) | 准确但耗内存 | 实时监控 |

**HeavyKeeper 论文参考**：
- 论文标题："HeavyKeeper: Tracking Accurate Quantiles in Data Streams"
- 核心思想：Count-Min Sketch + 最小堆 Top-K + 衰减机制 + 概率替换

#### 第三轮方案：HeavyKeeper 算法（当前方案✅）

**核心代码** ([HeavyKeeper.java](src/main/java/com/yuyuan/thumb/manager/cache/HeavyKeeper.java))：

```java
public class HeavyKeeper implements TopK {
    private final int k;           // Top-K 大小（监控前 100 个热 Key）
    private final int width;       // Count-Min Sketch 宽度（哈希表宽度 100000）
    private final int depth;       // Count-Min Sketch 深度（哈希次数 5）
    private final double decay;    // 衰减系数（0.92）
    private final int minCount;    // 最小记录阈值（出现 10 次才记录）
    
    private final Bucket[][] buckets;     // 多层计数桶
    private final PriorityQueue<Node> minHeap;  // 最小堆维护 Top-K
    
    @Override
    public AddResult add(String key, int increment) {
        byte[] keyBytes = key.getBytes();
        long itemFingerprint = hash(keyBytes);
        int maxCount = 0;

        // 1️⃣ 在多层 Count-Min Sketch 中更新计数
        for (int i = 0; i < depth; i++) {
            int bucketNumber = Math.abs(hash(keyBytes)) % width;
            Bucket bucket = buckets[i][bucketNumber];
            
            synchronized (bucket) {
                if (bucket.count == 0) {
                    // 空桶，直接写入
                    bucket.fingerprint = itemFingerprint;
                    bucket.count = increment;
                } else if (bucket.fingerprint == itemFingerprint) {
                    // 同一个 Key，累加计数
                    bucket.count += increment;
                } else {
                    // 不同 Key，概率衰减并可能替换
                    for (int j = 0; j < increment; j++) {
                        double d = lookupTable[bucket.count];
                        if (random.nextDouble() < d) {
                            bucket.count--;
                            if (bucket.count == 0) {
                                bucket.fingerprint = itemFingerprint;
                                bucket.count = increment - j;
                                break;
                            }
                        }
                    }
                }
                maxCount = Math.max(maxCount, bucket.count);
            }
        }

        // 2️⃣ 判断是否达到最小阈值
        if (maxCount < minCount) {
            return new AddResult(null, false, null);
        }
        
        // 3️⃣ 更新最小堆（Top-K 维护）
        synchronized (minHeap) {
            boolean isHot = false;
            String expelled = null;

            Optional<Node> existing = minHeap.stream()
                    .filter(n -> n.key.equals(key)).findFirst();
            
            if (existing.isPresent()) {
                // 已在堆中，更新计数
                minHeap.remove(existing.get());
                minHeap.add(new Node(key, maxCount));
                isHot = true;
            } else if (minHeap.size() < k || maxCount >= minHeap.peek().count) {
                // 新热 Key，尝试插入
                Node newNode = new Node(key, maxCount);
                if (minHeap.size() >= k) {
                    // 堆已满，挤出最冷的 Key
                    expelled = minHeap.poll().key;
                }
                minHeap.add(newNode);
                isHot = true;
            }

            return new AddResult(expelled, isHot, key);
        }
    }

    @Override
    public void fading() {
        // 定期衰减所有计数（每 20 秒执行一次）
        for (Bucket[] row : buckets) {
            for (Bucket bucket : row) {
                synchronized (bucket) {
                    bucket.count = bucket.count >> 1;  // 除以 2
                }
            }
        }
        
        // 同时衰减堆中元素的计数
        synchronized (minHeap) {
            PriorityQueue<Node> newHeap = new PriorityQueue<>(...);
            for (Node node : minHeap) {
                newHeap.add(new Node(node.key, node.count >> 1));
            }
            minHeap.clear();
            minHeap.addAll(newHeap);
        }
    }
}
```

**算法核心原理图解**：

```
输入流: [A, B, A, C, A, A, B, D, A, ...]

┌─────────────────────────────────────────────┐
│         Count-Min Sketch (5层 × 100000宽)   │
│                                              │
│  Layer 0: [A:4] [B:2] [C:1] [D:1] [...]     │
│  Layer 1: [A:3] [B:2] [C:1] [ ]   [...]     │
│  Layer 2: [A:4] [ ]   [C:1] [D:1] [...]     │
│  Layer 3: [A:3] [B:2] [ ]   [ ]   [...]     │
│  Layer 4: [A:4] [ ]   [ ]   [ ]   [...]     │
│                                              │
│  取最大值: A=4, B=2, C=1, D=1               │
└─────────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────────┐
│          Min-Heap (Top-K=100)               │
│                                              │
│         [A:4] ← 堆顶（最热的）              │
│        /    \                                │
│     [B:2]   [C:1]                           │
│      /                                      │
│   [D:1]                                     │
│                                              │
│  当新元素 E 计数 > 堆顶最小值时，替换堆顶    │
└─────────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────────┐
│           衰减机制（每20秒）                 │
│                                              │
│  所有计数 ÷ 2（右移一位）                   │
│  A:4 → 2 → 1 → 0（逐渐冷却）               │
│  冷数据自然淘汰，新热点有机会进入            │
└─────────────────────────────────────────────┘
```

**参数调优指南**：

| 参数 | 推荐范围 | 影响 |
|------|----------|------|
| k (Top-K大小) | 50-500 | 越大能跟踪更多热 Key，但内存和 CPU 开销增大 |
| width (宽度) | 10000-200000 | 越大哈希碰撞率越低，准确性越高 |

| decay (衰减系数) | 0.85-0.98 | 越接近 1 衰减越慢，历史数据权重越大 |
| minCount (最小阈值) | 5-50 | 越高过滤掉偶发访问，降低误判率 |

### 最终方案集成

**在 CacheManager 中的调用流程**：

```java
public Object get(String hashKey, String key) {
    String compositeKey = buildCacheKey(hashKey, key);

    // 1. 查询 L1 本地缓存
    Object value = localCache.getIfPresent(compositeKey);
    if (value != null) {
        hotKeyDetector.add(key, 1);  // 记录访问（即使命中也统计）
        return value;
    }

    // 2. 查询 L2 Redis
    Object redisValue = redisTemplate.opsForHash().get(hashKey, key);
    if (redisValue == null) return null;

    // 3. 判断是否为热 Key
    AddResult addResult = hotKeyDetector.add(key, 1);

    // 4. 热 Key 自动回填 L1
    if (addResult.isHotKey()) {
        localCache.put(compositeKey, redisValue);
        log.info("热Key回填L1: {} (访问次数={})", key, addResult);
    }

    return redisValue;
}

// 定时衰减（每 20 秒执行一次）
@Scheduled(fixedRate = 20, timeUnit = TimeUnit.SECONDS)
public void cleanHotKeys() {
    hotKeyDetector.fading();
}
```

### 设计逻辑

**为什么选择 HeavyKeeper 而非其他算法？**

1. **空间效率高**：Count-Min Sketch 仅需几位 bit 计数，相比 LFU 的 HashMap 节省 90%+ 内存
2. **支持衰减**：自动遗忘历史数据，适应访问模式变化
3. **概率保证**：虽然有一定误差率，但对热点识别场景足够准确
4. **实时性好**：O(1) 时间复杂度，适合高吞吐场景

**为什么不直接把所有数据都放入 L1？**
- L1 容量有限（1000 条），无法存储全部数据
- 进程内内存宝贵，不能无限扩展
- 冷数据放入 L1 浪费空间，挤出真正需要的热数据

## 8. Lua 脚本应用与原子性保障

### 问题背景

在高并发点赞场景下，"检查是否已点赞"和"写入点赞记录"这两个操作必须是原子的，否则会出现：

**竞态条件（Race Condition）示例**：
```
时间线:
T1: 用户A请求 "点赞博客101"
T2: 用户A请求 "点赞博客101"（网络重试或快速双击）

并发执行（无原子性保证）:
线程1: HEXISTS thumb:1 101 → 返回 0（未点赞）✓
线程2: HEXISTS thumb:1 101 → 返回 0（未点赞）✓  ← 问题！
线程1: HSET thumb:1 101 1 → 点赞成功 ✓
线程2: HSET thumb:1 101 1 → 重复点赞！✗
```

**后果**：
- 数据不一致（重复点赞）
- 博客点赞数虚高
- 取消点赞时逻辑混乱
- 用户投诉"点赞数不对"

### 技术调研

**Redis 原子性操作方案对比**：

| 方案 | 原子性 | 性能 | 复杂度 | 推荐 |
|------|--------|------|--------|------|
| **Lua 脚本** ⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐ | **⭐⭐⭐⭐⭐ 强烈推荐** |
| WATCH/MULTI/EXEC | ⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ 可选 |

| Redis + DB 事务 | ⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐ | ❌ 不推荐 |

#### 第三轮方案：Redis Lua 脚本（当前方案✅）

**核心代码** ([RedisLuaScriptConstant.java](src/main/java/com/yuyuan/thumb/constant/RedisLuaScriptConstant.java))：

```java
public class RedisLuaScriptConstant {

    /**
     * 点赞 Lua 脚本（用于 MQ 模式）
     * KEYS[1]: 用户点赞状态键 (thumb:{userId})
     * ARGV[1]: 博客 ID
     * 返回: -1=已点赞(失败), 1=成功
     */
    public static final RedisScript<Long> THUMB_SCRIPT_MQ = new DefaultRedisScript<>("""
            local userThumbKey = KEYS[1]
            local blogId = ARGV[1]
            
            -- 1. 判断是否已经点赞（原子性检查）
            if redis.call("HEXISTS", userThumbKey, blogId) == 1 then
                return -1  -- 已点赞，返回失败
            end
            
            -- 2. 添加点赞记录（原子性写入）
            redis.call("HSET", userThumbKey, blogId, 1)
            
            return 1  -- 成功
            """, Long.class);

    /**
     * 取消点赞 Lua 脚本
     */
    public static final RedisScript<Long> UNTHUMB_SCRIPT_MQ = new DefaultRedisScript<>("""
            local userThumbKey = KEYS[1]
            local blogId = ARGV[1]
            
            -- 1. 检查用户是否已点赞
            if redis.call("HEXISTS", userThumbKey, blogId) ~= 1 then
                return -1  -- 未点赞，返回失败
            end
            
            -- 2. 删除点赞标记
            redis.call("HDEL", userThumbKey, blogId)
            
            return 1  -- 成功
            """, Long.class);

    /**
     * 点赞 Lua 脚本（用于纯 Redis 模式，含临时计数）
     */
    public static final RedisScript<Long> THUMB_SCRIPT = new DefaultRedisScript<>("""
            local tempThumbKey = KEYS[1]       -- 临时计数键
            local userThumbKey = KEYS[2]       -- 用户状态键
            local userId = ARGV[1]
            local blogId = ARGV[2]
            
            -- 1. 检查是否已点赞
            if redis.call('HEXISTS', userThumbKey, blogId) == 1 then
                return -1
            end
            
            -- 2. 更新临时计数（用于后续同步到 DB）
            local hashKey = userId .. ':' .. blogId
            local oldNumber = tonumber(redis.call('HGET', tempThumbKey, hashKey) or 0)
            local newNumber = oldNumber + 1
            
            -- 3. 原子性写入：临时计数 + 状态标记
            redis.call('HSET', tempThumbKey, hashKey, newNumber)
            redis.call('HSET', userThumbKey, blogId, 1)
            
            return 1
            """, Long.class);
}
```

```java
                    log.error("点赞事件发送失败", ex);
                    return null;
                });

        return true;
    }
}
```

### Lua 脚本优势分析

**为什么选择 Lua？**

1. **原子性保证**：Redis 单线程执行 Lua，天然避免竞态条件
2. **减少网络 RTT**：多次 Redis 操作合并为一次网络往返
3. **服务端计算**：逻辑在 Redis 服务端执行，减少数据传输
4. **复用性强**：脚本可预加载（SCRIPT LOAD），后续通过 SHA1 调用

**性能对比**：

| 方案 | 网络往返次数 | 执行时间 | 原子性 |
|------|--------------|----------|--------|
| 分步执行（无保护） | 3 次 | 3ms | ❌ 不保证 |
| MULTI/EXEC | 1 次 | 2ms | ⚠️ 乐观锁可能失败 |

## 9. 虚拟线程应用与性能提升

### 问题背景

传统 Java 线程模型（Platform Thread）存在以下限制：

**具体问题**：
1. **线程成本高**：每个线程占用 ~1MB 栈内存，创建/销毁开销大
2. **并发数量受限**：单机最多创建几千个活跃线程（受限于内存和上下文切换开销）
3. **阻塞放大问题**：线程在等待 I/O 时被挂起，无法处理其他请求
4. **资源浪费严重**：大量时间花在线程调度而非实际业务处理

**实际数据对比**：

| 指标 | 平台线程 | 虚拟线程 |
|------|----------|----------|
| 创建耗时 | ~1ms | ~0.01ms |
| 栈内存 | ~1MB | ~几KB |
| 最大并发数 | ~2000 | **数百万** |
| 上下文切换 | 操作系统内核态 | **JVM 用户态（极快）** |
| 适合场景 | CPU 密集型 | **I/O 密集型 ⭐** |

### 技术调研

**官方文档**：
1. JEP 444: Virtual Threads（预览版）：https://openjdk.org/jeps/444
2. Spring Boot 3.2+ 虚拟线程支持：https://docs.spring.io/spring-boot/docs/current/reference/html/features.html#features.virtual-threads

**虚拟线程 vs 其他并发模型**：

| 模型 | 实现方式 | 适用场景 | 学习成本 |
|------|----------|----------|----------|
| **虚拟线程** ⭐ | JVM 层面 | I/O 密集型（Web 服务、数据库访问） | ⭐ 低 |
| CompletableFuture | JDK 库 | 异步编程 | ⭐⭐ 中 |
| RxJava/Reactor | 响应式编程 | 复杂异步流 | ⭐⭐⭐ 高 |
| Kotlin 协程 | 语言层面 | 全栈开发 | ⭐⭐ 中 |

### 设计逻辑

**为什么虚拟线程能带来如此大的提升？**

1. **轻量级**：创建成本几乎为零，可以"挥霍"般地为每个请求分配独立线程
2. **协作式调度**：I/O 操作时自动让出 CPU，无需操作系统介入
3. **无锁编程**：每个请求有自己的线程，无需共享状态（减少同步开销）
4. **更好的局部性**：CPU 缓存命中率提高（同一请求的数据在同一线程处理）

**适用场景判断**：

✅ **适合虚拟线程的场景**：
- Web 服务端（HTTP 请求处理）
- 数据库访问密集型应用
- RPC / HTTP 客户端调用
- 文件 I/O 操作

❌ **不适合虚拟线程的场景**：
- CPU 密集型计算（视频编码、密码学运算）
- synchronized 重代码（锁竞争仍会影响性能）
- Native 方法调用（可能 pin住载体线程）

### 注意事项与最佳实践

**1. 避免 synchronized（使用 ReentrantLock 替代）**

```java
// ❌ 不推荐：synchronized 会 pin 住载体线程
public synchronized void doSomething() { ... }

// ✅ 推荐：ReentrantLock 支持虚拟线程友好
private final ReentrantLock lock = new ReentrantLock();

public void doSomething() {
    lock.lock();
    try {
        // 业务逻辑
    } finally {
        lock.unlock();
    }
}
```

**2. 避免ThreadLocal滥用**

```java
// ❌ 问题：虚拟线程数量巨大，ThreadLocal可能导致内存泄漏
ThreadLocal<Connection> connectionHolder = ...

// ✅ 建议：使用Scope Values（JDK 21预览特性）替代
// 或者尽量减少ThreadLocal的使用
```

**3. 正确处理中断**

```java
// 虚拟线程更容易被中断，需要正确处理
try {
    Thread.sleep(1000);  // 可能抛出 InterruptedException
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();  // 恢复中断标志
    throw new RuntimeException("操作被取消", e);
}
```

## 10. 异步处理机制（Apache Pulsar 消息队列）

### 问题背景

在高并发点赞场景下，同步写入数据库会导致：

**具体问题**：
1. 每次点赞都要等数据库写入完成才返回（延迟 10-50ms）
2. 数据库连接池在高并发下耗尽（默认 10-20 个连接）
3. 数据库成为系统瓶颈（MySQL 单机写入 QPS 约 5000-10000）
4. 峰值流量时系统崩溃（无法削峰填谷）

**业务场景数据**：
- 峰值 QPS：50000+（明星发博瞬间）
- 平均 QPS：5000
- 数据库写入能力：8000 QPS
- 差距：峰值是平均值的 10 倍，远超数据库承受能力

### 技术调研

**消息队列选型对比**：

| 特性 | RabbitMQ | Apache Kafka | **Apache Pulsar** ⭐ | RocketMQ |
|------|----------|-------------|---------------------|----------|
| 吞吐量 | 中（万级） | 高（百万级） | **高（百万级）** | 高（十万级） |
| 延迟 | ms 级 | ms 级 | **ms 级** | ms 级 |
| 消息可靠性 | 高 | 高 | **极高（分层存储）** | 高 |
| 批量消费 | 支持 | 支持 | **原生支持** | 支持 |
| 多租户 | 不支持 | 不支持 | **原生支持** | 不支持 |
| 存储模型 | 队列 | 日志 | **Bookie（分层）** | 队列 |
| 运维复杂度 | 中 | 高 | **中等** | 高 |
| 社区活跃度 | 高 | 极高 | **高（快速增长）** | 中 |

**为什么选择 Pulsar？**

1. **存储计算分离**：Broker 无状态，易于扩缩容
2. **分层存储**：热数据在内存，冷数据在 S3/HDFS，成本更低
3. **强一致性**：支持 BookKeeper 的复制和 ACK 机制
4. **多租户支持**：适合 SaaS 化部署
5. **批量消费**：原生支持批量消息处理，减少 IO 次数

**官方文档**：
- Pulsar 官网：https://pulsar.apache.org/
- Spring Pulsar 文档：https://docs.spring.io/spring-pulsar/reference/

```java
@Service("thumbService")
@RequiredArgsConstructor
public class ThumbServiceMQImpl extends ServiceImpl<ThumbMapper, Thumb> implements ThumbService {

    private final RedisTemplate<String, Object> redisTemplate;
    private final PulsarTemplate<ThumbEvent> pulsarTemplate;

    @Override
    public Boolean doThumb(DoThumbRequest request, HttpServletRequest httpRequest) {
        User loginUser = userService.getLoginUser(httpRequest);
        Long loginUserId = loginUser.getId();
        Long blogId = request.getBlogId();
        String userThumbKey = RedisKeyUtil.getUserThumbKey(loginUserId);

        // 1️⃣ 执行 Lua 脚本，原子性写入 Redis
        long result = redisTemplate.execute(
                RedisLuaScriptConstant.THUMB_SCRIPT_MQ,
                List.of(userThumbKey),
                blogId
        );
        if (LuaStatusEnum.FAIL.getValue() == result) {
            throw new RuntimeException("用户已点赞");
        }

        // 2️⃣ 构建事件对象
        ThumbEvent event = ThumbEvent.builder()
                .blogId(blogId)
                .userId(loginUserId)
                .type(ThumbEvent.EventType.INCR)
                .eventTime(LocalDateTime.now())
                .build();

        // 3️⃣ 异步发送到 Pulsar（非阻塞！）
        pulsarTemplate.sendAsync("thumb-topic", event)
                .exceptionally(ex -> {
                    // 发送失败则补偿回滚 Redis
                    redisTemplate.opsForHash().delete(userThumbKey, blogId.toString());
                    log.error("点赞事件发送失败: userId={}, blogId={}", loginUserId, blogId, ex);
                    return null;
                });

        // 4️⃣ 立即返回成功（总耗时 < 5ms）
        return true;
    }
}
```

**Consumer 实现** ([ThumbConsumer.java](src/main/java/com/yuyuan/thumb/listener/thumb/ThumbConsumer.java))：

```java
@Service
@RequiredArgsConstructor
@Slf4j
public class ThumbConsumer {

    private final BlogMapper blogMapper;
    private final ThumbService thumbService;

    /**
     * 批量消费配置
     * - topics: thumb-topic
     * - batch: true（启用批量消费）
     * - subscriptionType: Shared（共享订阅，支持并行消费）
     * - maxNumMessages: 1000（每批最大消息数）
     * - timeout: 10s（超时即触发消费）
     */
    @PulsarListener(
            subscriptionName = "thumb-subscription",
            topics = "thumb-topic",
            schemaType = SchemaType.JSON,
            batch = true,
            subscriptionType = SubscriptionType.Shared,
            negativeAckRedeliveryBackoff = "negativeAckRedeliveryBackoff",
            ackTimeoutRedeliveryBackoff = "ackTimeoutRedeliveryBackoff",
            deadLetterPolicy = "deadLetterPolicy"
    )
    @Transactional(rollbackFor = Exception.class)
    public void processBatch(List<Message<ThumbEvent>> messages) {
        log.info("收到批量消息: {} 条", messages.size());

        Map<Long, Long> countMap = new ConcurrentHashMap<>();
        List<Thumb> thumbsToInsert = new ArrayList<>();

        // 1️⃣ 提取有效事件
        List<ThumbEvent> events = messages.stream()
                .map(Message::getValue)
                .filter(Objects::nonNull)
                .toList();

        // 2️⃣ 按 (userId, blogId) 分组，取最终状态
        Map<Pair<Long, Long>, ThumbEvent> latestEvents = events.stream()
                .collect(Collectors.groupingBy(
                        e -> Pair.of(e.getUserId(), e.getBlogId()),
                        Collectors.collectingAndThen(
                                Collectors.toList(),
                                list -> {
                                    // 按时间排序，奇数次=点赞，偶次数=取消
                                    list.sort(Comparator.comparing(ThumbEvent::getEventTime));
                                    if (list.size() % 2 == 0) return null;  // 偶数次=最终取消
                                    return list.get(list.size() - 1);     // 奇数次=最终点赞
                                }
                        )
                ));

        // 3️⃣ 分类处理
        LambdaQueryWrapper<Thumb> deleteWrapper = new LambdaQueryWrapper<>();
        AtomicReference<Boolean> needDelete = new AtomicReference<>(false);

        latestEvents.forEach((userBlogPair, event) -> {
            if (event == null) return;
            
            if (event.getType() == ThumbEvent.EventType.INCR) {
                // 点赞：准备插入
                countMap.merge(event.getBlogId(), 1L, Long::sum);
                thumbsToInsert.add(new Thumb(event.getUserId(), event.getBlogId()));
            } else {
                // 取消点赞：准备删除
                needDelete.set(true);
                deleteWrapper.or()
                        .eq(Thumb::getUserId, event.getUserId())
                        .eq(Thumb::getBlogId, event.getBlogId());
                countMap.merge(event.getBlogId(), -1L, Long::sum);
            }
        });

        // 4️⃣ 批量写入数据库
        if (needDelete.get()) {
            thumbService.remove(deleteWrapper);  // 批量删除
        }
        batchUpdateBlogs(countMap);              // 批量更新博客点赞数
        batchInsertThumbs(thumbsToInsert);       // 批量插入点赞记录
    }

    public void batchUpdateBlogs(Map<Long, Long> countMap) {
        if (!countMap.isEmpty()) {
            blogMapper.batchUpdateThumbCount(countMap);  // 自定义批量 SQL
        }
    }

    public void batchInsertThumbs(List<Thumb> thumbs) {
        if (!thumbs.isEmpty()) {
            thumbService.saveBatch(thumbs, 500);  // MyBatis-Plus 批量插入，每批 500 条
        }
    }
}
```

```java
                BatchReceivePolicy.builder()
                        .maxNumMessages(1000)             // 每批最多 1000 条
                        .timeout(10000, TimeUnit.MILLISECONDS)  // 超时 10 秒
                        .build()
        );
    }

    // NACK 重试策略（消费者主动拒绝消息）
    @Bean
    public RedeliveryBackoff negativeAckRedeliveryBackoff() {
        return MultiplierRedeliveryBackoff.builder()
                .minDelayMs(1000)     // 初始延迟 1 秒
                .maxDelayMs(60000)    // 最大延迟 60 秒
                .multiplier(2)        // 每次倍增
                .build();
    }

    // ACK 超时重试策略（消费者未及时确认）
    @Bean
    public RedeliveryBackoff ackTimeoutRedeliveryBackoff() {
        return MultiplierRedeliveryBackoff.builder()
                .minDelayMs(5000)      // 初始延迟 5 秒
                .maxDelayMs(300_000)   // 最大延迟 5 分钟
                .multiplier(3)         // 每次乘以 3
                .build();
    }

    // 死信队列策略
    @Bean
    public DeadLetterPolicy deadLetterPolicy() {
        return DeadLetterPolicy.builder()
                .maxRedeliverCount(3)                  // 最多重试 3 次
                .deadLetterTopic("thumb-dlq-topic")    // 死信主题
                .build();
    }
}
```

### 关键设计：消息去重与幂等性

**问题**：用户快速双击点赞按钮，会产生两个 INCR 事件；或者先点赞后取消，产生 INCR + DECR 两个事件。

**解决方案**：在 Consumer 端按 `(userId, blogId)` 分组，只保留最终状态。

```java
// 按 (userId, blogId) 分组
Map<Pair<Long, Long>, ThumbEvent> latestEvents = events.stream()
    .collect(Collectors.groupingBy(
        e -> Pair.of(e.getUserId(), e.getBlogId()),
        Collectors.collectingAndThen(
            Collectors.toList(),
            list -> {
                // 按时间排序
                list.sort(Comparator.comparing(ThumbEvent::getEventTime));
                // 奇数次 = 最终点赞，偶次数 = 最终取消
                if (list.size() % 2 == 0) return null;
                return list.get(list.size() - 1);
            }
        )
    ));
```

**示例**：
```
输入事件流（时间顺序）：
[INCR(user1, blog1), INCR(user1, blog1), DECR(user1, blog1)]

分组后 (user1, blog1):
1. INCR (t=10:00:00)
2. INCR (t=10:00:01)  ← 重复！
3. DECR (t=10:00:02)

处理结果：
- 事件数 = 3（奇数）→ 最终状态 = 最后一个事件 = DECR
- 结论：用户最终取消了点赞，不插入记录
```

### 设计逻辑

**为什么选择异步而非同步？**

1. **用户体验优先**：快速响应让用户感觉系统流畅
2. **削峰填谷**：将瞬时高峰转化为平稳的批量处理
3. **解耦**：点赞逻辑与持久化解耦，独立扩缩容
4. **容错**：消息队列提供重试、死信、补偿机制

**为什么选择批量消费而非逐条消费？**

| 方案 | 数据库 IO 次数（1000条消息） | 吞吐量 |
|------|-------------------------------|--------|
| 逐条消费 | 1000 次 INSERT + 1000 次 UPDATE | 低 |
| **批量消费** ⭐ | **2 次（1批 INSERT + 1批 UPDATE）** | **高 500x** |
