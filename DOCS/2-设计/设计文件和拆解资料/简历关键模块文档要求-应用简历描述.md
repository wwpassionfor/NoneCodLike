## 模块1：SpringBoot3 核心业务架构 + 数据库设计
### 对应解决痛点
上线慢、数据不一致、多租户数据混乱、API对接困难
### 具象问题场景
1. 校园：多个社团同时用系统做活动，数据互相串扰；
2. 企业：不同商家的优惠券、订单数据隔离失败，查询报错；接口无文档，前端/多渠道对接效率极低。
### 可视化方案
**架构图**（前后端分离三层架构）
`客户/运营端 → 零代码配置前端 → SpringBoot3网关层 → 业务服务层 → MyBatis-Plus → MySQL/TiDB`
**流程图**（核心角色）
`运营配置活动 → API接口请求 → 全局跨域处理 → 数据序列化 → 多租户数据隔离 → 数据库CRUD`
### 分步骤逻辑解析
1. 搭建项目基础框架，统一技术栈规范；
2. 解决全局通用问题（跨域、精度丢失）；
3. 设计核心业务数据表，实现多租户隔离；
4. 索引优化，提升查询速度；
5. 自动生成API文档，支撑多渠道快速发布。
### 技术实现+关键代码思路
```
技术栈：SpringBoot3 + MyBatis-Plus + Lombok + Hutool + Knife4j
1. 全局CORS配置：解决H5/小程序/APP跨域访问问题
2. Jackson配置：修复Long类型精度丢失（订单ID/活动ID）
3. 核心表：activity(活动)、coupon(优惠券)、user_behavior(用户行为)、order(订单)
4. 多租户：tenant_id字段+联合索引，实现商家数据物理隔离
5. Knife4j：自动生成在线API文档，零代码平台快速对接
```
### 实践成果+技术文档
- 成果：多租户数据100%隔离，接口对接效率提升70%，修复序列化/跨域线上bug
- 文档：《项目技术栈规范文档》《核心数据库表设计文档》《API接口文档》

## 模块2：多级缓存 + 热点数据治理
### 对应解决痛点
系统峰值崩溃、接口响应慢、缓存雪崩/击穿
### 具象问题场景
1. 校园：校庆秒杀1分钟内3000人访问，接口卡顿超时；
2. 企业：电商爆款优惠券，大量请求直接打穿数据库。
### 可视化方案
**两级缓存架构图**
`用户请求 → Nginx → Caffeine本地缓存(一级) → Redis Hash(二级) → 数据库`
**热点探测流程图**
`请求流量 → HeavyKeeper算法统计 → 识别热点活动/优惠券 → 自动缓存预热 → TTL自动过期`
### 分步骤逻辑解析
1. 搭建本地+分布式两级缓存架构；
2. 针对秒杀/优惠券等热点数据做缓存适配；
3. 接入热点探测算法，自动识别高流量数据；
4. 管控缓存生命周期，防止故障；
5. 压测验证缓存命中率与响应速度。
### 技术实现+关键代码思路
```
1. Caffeine：本地缓存，支撑单机高并发读
2. Redis Hash：存储活动配置、优惠券库存，支持批量查询
3. HeavyKeeper：实时统计热点Key，避免无效缓存
4. 策略：缓存预热、超时降级、互斥锁解决缓存击穿
```
### 实践成果+技术文档
- 成果：接口响应时间降低62%，缓存命中率94%+，杜绝缓存故障
- 文档：《多级缓存设计文档》《热点数据治理方案》《缓存压测报告》

## 模块3：消息队列削峰 + 异步批量处理
### 对应解决痛点
峰值崩溃、数据库IO过高、系统吞吐量低、同步阻塞
### 具象问题场景
1. 校园：线上投票/集赞，每秒上千条行为数据写入，数据库卡死；
2. 企业：秒杀下单、消息通知同步执行，系统直接宕机。
### 可视化方案
**异步削峰架构图**
`用户行为(下单/集赞/抽奖) → Apache Pulsar消息队列 → 批量消费组件 → 数据统计/通知/入库`
**流程图（核心角色）**
`用户触发行为 → 同步返回结果 → 异步发送消息 → 队列缓冲 → 批量处理 → 数据库写入`
### 分步骤逻辑解析
1. 集成消息队列，解耦核心业务与非核心业务；
2. 将秒杀、集赞、统计等高频操作异步化；
3. 批量消费消息，降低数据库压力；
4. 保证消息可靠性，避免丢失；
5. 压测提升系统吞吐量。
### 技术实现+关键代码思路
```
组件：Apache Pulsar
1. 主题拆分：秒杀主题、行为日志主题、通知主题
2. 批量消费：每次消费100条，合并SQL执行
3. 异步处理：不阻塞前端响应，提升用户体验
```
### 实践成果+技术文档
- 成果：系统吞吐量提升4.3倍，单机QPS 5000+，数据库IO降低70%
- 文档：《消息队列设计文档》《异步批量处理方案》《高并发压测报告》

## 模块4：分布式数据库 TiDB 存储扩容
### 对应解决痛点
单机数据库瓶颈、数据不一致、分库分表复杂、百万数据存储困难
### 具象问题场景
1. 校园：多场活动累计10万+用户行为数据，单机MySQL卡顿；
2. 企业：多商家订单/活动数据量暴涨，单机库无法扩容。
### 可视化方案
**分布式存储架构图**
`应用服务 → MyBatis-Plus → TiDB分布式数据库 → 自动分片 → 多节点存储`
**流程图**
`数据写入 → TiDB自动分片 → 分布式事务 → 多节点持久化 → 高可用查询`
### 分步骤逻辑解析
1. 从单机MySQL迁移至TiDB；
2. 实现数据自动分片，无需手动分库分表；
3. 保证分布式事务一致性；
4. 支撑水平扩容，适配百万级数据。
### 技术实现+关键代码思路
```
1. 无缝迁移：保留MyBatis-Plus语法，无代码入侵
2. 自动分片：按租户ID/活动ID分片
3. 分布式事务：保证订单/优惠券/库存数据一致
```
### 实践成果+技术文档
- 成果：支撑百万级数据存储，无分库分表复杂度，数据一致性100%
- 文档：《分布式数据库迁移方案》《TiDB运维文档》《数据容量规划报告》

## 模块5：流量管控 + 高可用降级保护
### 对应解决痛点
流量过载、服务宕机、核心功能不可用、极端场景崩溃
### 具象问题场景
1. 校园：秒杀流量突增，Redis宕机导致整个系统不可用；
2. 企业：大促流量超阈值，非核心功能拖垮核心秒杀。
### 可视化方案
**高可用架构图**
`Redis主从+哨兵 → 多级降级组件 → 流量管控 → 核心服务保障`
**降级流程图**
`流量检测 → 超阈值/服务故障 → 执行降级策略 → 关闭非核心功能 → 保障秒杀/优惠券`
### 分步骤逻辑解析
1. 搭建Redis高可用架构（主从+哨兵）；
2. 设计三级降级策略；
3. 实时流量监控，自动触发降级；
4. 故障恢复后自动回切。
### 技术实现+关键代码思路
```
1. Redis高可用：主从复制，哨兵自动故障切换
2. 降级策略：
   - 缓存失效：切换本地缓存 / 直查TiDB
   - 流量超限：关闭排行榜、分享统计等非核心功能
3. 限流：Redis+Lua脚本实现分布式限流
```
### 实践成果+技术文档
- 成果：极端流量下核心服务100%可用，无全盘崩溃
- 文档：《高可用设计方案》《服务降级与限流文档》《故障应急手册》
