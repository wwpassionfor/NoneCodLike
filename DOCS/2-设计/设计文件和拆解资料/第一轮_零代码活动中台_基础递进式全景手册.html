"为中小企业、高校、品牌方提供一套不用写代码就能配置上线的营销活动系统，同时能扛住高并发、能防作弊、能保证数据不出错。"

1.2 从校园痛点出发，看懂商业需求

📍 校园起步场景（学生团队时期）

学生会要做"防诈骗知识测评"，社团要做"招新集赞"，团委要做"校庆秒杀"。每次找技术团队，都要重新开发页面、重新写接口、重新调数据库，一个活动上线要7天。活动只办3天，投入产出严重失衡。更惨的是，活动开始时几百人同时点击，系统直接白屏崩溃。

📍 商业化场景（推广至中小企业后）

校外电商客户要做"双十一秒杀"，品牌方要做"答题领券"。客户没有技术团队，等不起2周开发周期；活动开启时几万用户涌入，系统必须不能崩；更要防止羊毛党批量薅券、防止库存超卖导致客诉。

🔴 问题本质：传统方式的"四座大山"

大山校园表现企业表现根因
上线慢7天开发一个投票页2周开发一个秒杀页每次重新造轮子，没有标准化模板
峰值崩溃300人同时点击就白屏10万人涌入系统宕机请求直接打数据库，没有缓冲层
风控弱刷票、机器点赞无法识别羊毛党批量注册薅券没有设备识别和行为分析能力
数据不一致库存显示有货实际已空订单超卖、优惠券重复领取并发扣减没有原子保护

2.2 逐个组件大白话

🟢 SpringBoot3 —— 系统的"骨架"

比喻：像建筑的钢结构框架，把所有零件（缓存、数据库、消息队列）规范地组装在一起。

在本项目中的作用：

  • 提供HTTP接口（学生点击链接后，请求就是打到SpringBoot服务）
  • 管理业务逻辑（活动怎么创建、用户怎么参与、分数怎么算）
  • 连接所有中间件（通过配置文件把Redis、TiDB、Pulsar串起来）

为什么选3.x而不是2.x：支持JDK17，性能更好，响应式编程支持更强，长期维护。

🟢 MyBatis-Plus —— 数据库操作的"翻译官"

比喻：你写Java代码说"查订单"，MyBatis-Plus帮你翻译成MySQL/TiDB能听懂的SQL语句。

在本项目中的作用：

  • 不用手写大量SQL，用Java代码就能完成增删改查
  • 自动处理"多租户"（不同客户的数据自动隔离）
  • 支持Lambda查询，代码更简洁

🟢 Redis Cluster —— 系统的"高速记事本"

比喻：数据库像图书馆（存书多但找书慢），Redis像办公桌上的便签纸（记的东西少但看一眼就知道）。

在本项目中的作用：

  • 缓存热点数据：活动配置、库存数量、排行榜，避免每次都查数据库
  • 分布式锁：防止多个用户同时抢到同一件商品（超卖）
  • 限流计数：记录某IP/某用户点击了多少次，超过就拦截
  • 会话存储：用户登录态存在Redis，多服务器共享

为什么用Cluster（集群）而不是单机：单机Redis挂了缓存全丢，集群多节点互相备份，容量也更大。

🟢 Caffeine —— 应用内的"贴身便签"

比喻：Redis是"办公室公共便签墙"，Caffeine是"你自己桌上的小本本"，伸手就能看，不用起身。

在本项目中的作用：

  • 存在应用服务器内存里，读取速度极快（<1毫秒）
  • 存放"最热"的数据（比如当前正在进行的秒杀活动配置）
  • 减轻Redis压力（很多请求在本地就拦截了，不用走网络）

🟢 Apache Pulsar —— 系统的"排队叫号机"

比喻：医院高峰期病人太多，医生看不过来，先取号排队，医生按顺序叫号，不会挤爆诊室。

在本项目中的作用：

  • 学生集中交卷、秒杀集中下单时，请求不直接写数据库，而是先"取号"进队列
  • 后端服务按固定速度"叫号"处理，把脉冲式高峰削成平缓流量
  • 支持批量处理（一次叫500个号一起看），效率更高

🟢 TiDB —— 能自动扩容的"智能仓库"

比喻：普通MySQL像一间固定大小的仓库，满了就得人工搬东西换大房子；TiDB像乐高积木仓库，不够用了自动拼接新的积木块扩容。

在本项目中的作用：

  • 存所有业务数据（活动、订单、用户行为、优惠券）
  • 数据自动分片（不用人工分库分表），按活动ID或用户ID自动分散到不同节点
  • 支持分布式事务（跨多个节点的操作也能保证ACID）
  • 同时支持实时分析（一份数据既能查单笔订单，也能做全校统计）

🟢 Knife4j —— API文档的"自动打印机"

比喻：以前接口文档靠程序员手写Word，经常过时；Knife4j像连接在代码上的打印机，代码怎么写，文档就怎么出，实时更新。

在本项目中的作用：前端H5、小程序、第三方渠道对接时，直接在线看文档、在线调试，不用口头对齐字段。

第三章：核心库表设计理解 —— 数据是怎么流转的？

3.1 为什么需要理解库表？

架构图里的每个方块（活动服务、订单服务、风控服务），本质上都是在操作这些表。看懂表的设计，就能看懂数据怎么流、为什么不会乱。

3.2 四大核心领域表

📋 活动配置表（activity）—— "活动的户口本"

存什么：每个活动的基本信息（名称、时间、类型、规则、参与条件）。

关键字段理解：

字段作用为什么重要
tenant_id归属哪个客户/租户客户A的活动，客户B绝对看不到
activity_type秒杀/集赞/抽奖/测评/拼团决定前端渲染什么模板
rule_configJSON格式的规则（如"每人限3次"）零代码配置的核心：规则存在JSON里，不用改代码
status未开始/进行中/已结束定时任务靠它自动开关活动
start_time / end_time起止时间缓存TTL、定时任务都依赖它

📋 用户参与记录表（user_behavior）—— "谁干了什么"

存什么：每个用户在每个活动里的每一次操作（点击、答题、点赞、下单）。

关键字段理解：

字段作用为什么重要
user_id用户身份关联OneID，跨渠道识别同一人
activity_id参与哪个活动聚合统计"某活动参与人数"
behavior_type行为类型（vote/like/exam/order）区分是投票还是下单
device_fingerprint设备指纹风控识别同一设备多账号
create_time发生时间时序分析、防刷（如1秒内点10次）

📋 订单表（order）—— "交易凭证"

存什么：秒杀、拼团等涉及交易的订单信息。

关键字段理解：

字段作用为什么重要
order_id订单唯一编号（Long型）传给前端时必须转String，防精度丢失
activity_id / sku_id关联活动/商品统计"某商品卖出多少"
status待支付/已支付/已取消超时未支付要自动回退库存
version乐观锁版本号并发扣减时防止覆盖（防超卖核心）

📋 优惠券表（coupon + coupon_record）—— "券的生命周期"

存什么：券模板（有什么券）和券领取记录（谁领了谁用了）。

关键字段理解：

字段作用为什么重要
total_stock / remain_stock总库存/剩余库存秒杀场景下必须原子扣减
user_limit每人限领几张防薅羊毛的第一道规则
valid_start / valid_end有效期过期自动失效，定时清理
record.status已领取/已使用/已过期防止重复使用和状态不一致

4.3 分步骤逻辑解析

1
运营端拖拽配置（可视化表单）
后台页面提供组件库：基础信息（名称/时间）、内容组件（视频上传/题目编辑器）、规则组件（参与次数/及格线/切屏检测）、奖品组件（优惠券/实物/证书）、通知组件（微信/QQ/短信）。运营人员拖拽填写，前端实时预览效果。
2
系统生成JSON规则包
表单提交后，后端将配置转化为结构化JSON。例如测评活动的rule_config：
{ "type": "exam", "passScore": 80, "maxAttempts": 3, "questions": [ {"id": 1, "type": "single", "answer": "A", "score": 10}, {"id": 2, "type": "multi", "answer": "AB", "score": 10} ], "antiCheating": { "screenSwitchLimit": 3, "minAnswerTime": 30 }, "notification": ["wechat", "qq"] }
3
规则持久化与缓存预热
JSON存入activity表的rule_config字段，同时生成标准activity记录（含tenant_id隔离）。发布瞬间，系统将该活动配置写入Redis Hash和Caffeine本地缓存，确保用户访问时无需查库即可获取规则。
4
前端动态渲染（模板引擎）
用户点击活动链接，前端根据activity_type加载对应模板（exam.html / seckill.html / lottery.html）。模板通过AJAX请求获取活动JSON规则，动态渲染页面：如果是测评，渲染题目和倒计时；如果是秒杀，渲染库存和抢购按钮。同一套后端，不同活动类型走不同前端模板。
5
用户行为实时校验（规则引擎执行）
用户提交答案/点击抢购时，后端加载JSON规则实时校验：是否超过maxAttempts？答题时间是否<minAnswerTime？切屏次数是否>screenSwitchLimit？所有规则在内存中解析执行，无需改代码即可适配不同活动的风控强度。

4.4 为什么这能实现"零代码"？

🔴 传统方式 vs 零代码方式对比

环节传统方式零代码方式
需求传递运营写Word → 技术理解偏差 → 返工运营自己在后台配置，所见即所得
规则变更"每人限3次改5次" → 提需求 → 改代码 → 发版后台改数字 → 实时生效

💡 核心认知：零代码不是"没有代码"，而是把代码变成了"可配置的规则"。技术团队把活动的共性逻辑（怎么算分、怎么扣库存、怎么发通知）写成模板引擎，运营人员只需要填参数，不用关心代码怎么实现。

第五章：并发问题识别 —— 为什么需要分布式锁、消息队列、多级缓存？

5.1 先建立直觉：什么是并发问题？

📍 场景：超卖（库存扣减竞态条件）

库存只剩1件，用户A和用户B同时点击"立即抢购"。

  • 时刻1：A查库存 = 1，B查库存 = 1（同时读到）
  • 时刻2：A扣减库存 → 0，B扣减库存 → -1（B也成功了！）
  • 结果：卖出去2件，实际只有1件 → 超卖

📍 场景：重复提交（网络延迟+用户焦虑）

学生交卷后页面转圈3秒没反应，他以为没成功，又点了一次提交。

  • 请求1：正在处理中（还没写库）
  • 请求2：又进来了，系统不知道这是重复请求
  • 结果：数据库里有两条一模一样的交卷记录 → 重复计分

📍 场景：缓存击穿（热点Key集体失效）

活动开始前，运营把所有商品缓存TTL设为10分钟。10分钟后所有缓存同时过期。

  • 时刻T：1000个缓存Key同时失效
  • 10000个用户请求同时打到数据库
  • 数据库瞬间崩溃 → 缓存击穿引发雪崩

5.3 三大并发武器的原理

🔒 分布式锁 —— "排队买票，一个一个来"

解决什么问题：多个用户同时修改同一份数据（如库存）。

怎么实现：利用Redis的SETNX命令（Set if Not Exists）。谁先在Redis里占住一个Key（如"lock:stock:888"），谁就有权利扣减库存；其他人发现Key已被占用，就等待或返回"抢购中"。处理完后删除Key，下一个人进来。

在本项目中的应用：秒杀库存扣减、优惠券领取、答题次数限制。

// 伪代码：Redis分布式锁防止超卖 function deductStock(skuId) { lockKey = "lock:stock:" + skuId; // 尝试占坑（有效期5秒，防止死锁） ok = redis.set(lockKey, "1", "NX", "EX", 5); if (!ok) return "系统繁忙，请重试"; try { stock = redis.get("stock:" + skuId); if (stock > 0) { redis.decr("stock:" + skuId); // 原子扣减 return "抢购成功"; } else { return "库存已空"; } } finally { redis.del(lockKey); // 释放锁 } }

📨 消息队列削峰 —— "高峰期排队，低峰期处理"

解决什么问题：几千人同时交卷/下单，数据库INSERT不过来。

怎么实现：请求不直接写数据库，而是写入Pulsar队列（像医院的叫号机）。队列能扛住10万TPS的写入。后端消费服务按自己的节奏（比如每秒处理5000条）从队列里取数据批量写入数据库。这样数据库永远只承受匀速流量，不会被打死。

在本项目中的应用：秒杀订单创建、集赞记录、抽奖记录、测评交卷、消息通知。

// 伪代码：削峰流程 function submitExam(userId, answers) { // 1. 立即校验（参数、风控） if (!riskCheck(userId)) return "存在异常"; // 2. 写入队列（极快，毫秒级） pulsar.send("exam-submit", {userId, answers, time: now()}); // 3. 立即返回用户（不用等数据库） return "提交成功，评分中..."; } // 后端消费（批量处理） function consumeBatch(messages) { // 一次拿500条，合并成一条INSERT sql = "INSERT INTO exam_record VALUES " + join(messages); db.execute(sql); }

💾 多级缓存 —— "越近越快，层层拦截"

解决什么问题：80%的请求其实在查同样的热点数据（如活动配置、库存），每次都查数据库太浪费。

怎么实现：设计三级拦截：浏览器本地缓存（最前）→ CDN边缘缓存 → Caffeine应用缓存 → Redis分布式缓存 → 数据库（最后）。请求尽可能在前面被拦住，只有极少数落到数据库。

在本项目中的应用：活动规则查询、秒杀库存查询、排行榜查询、题目内容查询。

// 伪代码：多级缓存读取 function getActivity(id) { // L1: 本地缓存（<1ms） data = caffeine.get(id); if (data) return data; // L2: Redis（<5ms） data = redis.get("activity:" + id); if (data) { caffeine.put(id, data); // 回填本地 return data; } // L3: 数据库（<50ms） data = db.query("SELECT * FROM activity WHERE id=?", id); redis.set("activity:" + id, data, 300); // 回填Redis caffeine.put(id, data); // 回填本地 return data; }
💡 核心认知：并发问题的本质不是"人多"，而是"多人同时操作同一份数据，且操作之间没有隔离"。分布式锁解决"写冲突"，消息队列解决"写压力"，多级缓存解决"读压力"。三者叠加，系统才能从"一崩全崩"进化为"高并发下仍稳定"。

第六章：安全问题理解 —— 风控层怎么防刷票、防薅羊毛？

6.1 安全威胁的真实场景

📍 场景1：刷票（歌手大赛投票）

某选手粉丝群组织刷票：10台手机连同一WiFi，每秒钟自动点击投票按钮。传统系统只看"一个账号投一次"，但10个账号就是10台设备的真实操作吗？不是，背后是1个人在操作。

📍 场景2：薅羊毛（优惠券领取）

羊毛党使用接码平台注册100个手机号，批量领取优惠券后转卖。系统只看到"100个不同手机号"，但背后是同一批设备、同一IP段、甚至相同的鼠标点击轨迹。

📍 场景3：考试作弊（防诈骗测评）

学生A把账号给代考机构，代考机构在异地登录，5分钟答完所有题且正确率100%。或者学生B切屏搜索答案，系统没有任何感知。

6.3 三层防御的具象逻辑

🛡️ L1 入口层：规则硬拦截（毫秒级）

手段：

  • IP限频：同一IP 1秒内最多5次请求，超限直接拒绝（防脚本高频刷）
  • 账号限次：同一账号每天最多投3票/领2张券（防单账号重复薅）
  • 设备黑名单：已知作弊设备直接拦截（基于历史风控标记）

在本项目中的应用：秒杀下单前的令牌桶限流、投票接口的IP+账号双重限频。

🛡️ L2 行为层：机器 vs 真人（百毫秒级）

手段：

  • 设备指纹：采集Canvas指纹、字体列表、电池状态、屏幕分辨率，生成唯一设备ID。同一设备登录多个账号 = 高度可疑
  • 行为序列：记录鼠标移动轨迹、点击间隔、滑动速度。机器操作通常是"直线移动、固定间隔、无抖动"
  • 切屏检测：前端JS监听visibilitychange事件，切屏次数超过阈值自动标记
  • 答题时长异常：正常学习视频10分钟，答题5分钟；若打开页面30秒就交卷且全对 = 可疑

在本项目中的应用：测评活动的切屏检测、投票活动的设备指纹绑定、抽奖活动的行为验证码（滑动拼图）。

🛡️ L3 数据层：团伙与模式分析（分钟级）

手段：

  • 答案相似度：SimHash算法比对答卷向量，相似度>85%视为"雷同卷"
  • 团伙检测：Louvain图算法分析"IP聚集+答题时间聚集+答案一致"的社群，揪出组织性作弊
  • 时序异常：某账号在1秒内完成"打开页面→答题→交卷"全流程，明显非人类操作

在本项目中的应用：考试结束后批量扫描雷同卷、活动期间实时检测团伙刷票。
